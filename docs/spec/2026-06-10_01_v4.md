## §0 — Validaciones de input adicionales

Guardias de negocio que pertenecen al schema Joi, no a la capa de servicio. El servicio debe poder asumir que los datos de entrada ya son válidos; las violaciones de límites son errores de cliente (4xx), no errores de dominio.

### §0.1 — Límite de pertenencia a grupos en PATCH /api/v1/users/:id

**Decisión del usuario:** un usuario no puede pertenecer a más de 10 grupos. Si el body de `PATCH /api/v1/users/:id` incluye un array `groups` con más de 10 elementos, la request debe rechazarse antes de llegar al handler.

**Qué cambia:** añadir `.max(10)` a la definición del campo `groups` en `updateUserByAdminSchema` (y en `createUserSchema` por consistencia — no tiene sentido crear un usuario ya en violación del límite).

**Archivo:** `src/api/users/schemas/user.schema.js`

Antes (línea 6 — definición base del field):
```js
const groups = Joi.array().items(Joi.string());
```

Después:
```js
const groups = Joi.array().items(Joi.string()).max(10);
```

Dado que la variable `groups` se reutiliza en `createUserSchema` (línea 26) y `updateUserByAdminSchema` (línea 39), añadir `.max(10)` a la definición base aplica el límite en ambos schemas simultáneamente.

**Mensaje de error que retorna Joi:** `"groups" must contain less than or equal to 10 items`

El `validatorHandler` transforma esto en un `boom.badRequest(error.message)`, que el `errorHandler` responde como `400 Bad Request` con cuerpo JSON:
```json
{
  "statusCode": 400,
  "error": "Bad Request",
  "message": "\"groups\" must contain less than or equal to 10 items"
}
```

**Por qué aquí y no en el servicio:** la validación de límites de input pertenece a la frontera del sistema (schema Joi), no a la capa de dominio. El servicio recibe datos ya validados; si detectara el límite, tendría que lanzar un error de negocio mezclado con responsabilidad de validación de entrada. Además, `GroupService.update()` — que también sincroniza `User.groups` via batch — no pasa por este schema; no se duplica la guardia allá porque el caso de uso de sync es distinto al caso de input directo del cliente.

**Relación con §1.4:** `array-contains-any` acepta máximo 10 valores. El límite de 10 grupos por usuario garantiza que el query de `GET /api/v1/redirects` para usuarios con grupos nunca exceda ese límite.

> **ADVERTENCIA:** si este límite se incrementa por encima de 10, el query `array-contains-any` de `GET /api/v1/redirects` superará el máximo de Firestore. Ver runbook `docs/runbooks/increase-group-limit.md` para la solución de batching requerida.

**Archivo:** `src/utils/crud.service.js` línea 74.

## §1 — Refactorización de FireStoreAdapter

### Problema

El objetivo declarado por el usuario es "optimizar las consultas para reducir costos de GCP". En Firestore se cobra por:
- **Lecturas de documento:** $0.06 por 100K lecturas
- **Escrituras de documento:** $0.18 por 100K escrituras
- **Borrados de documento:** $0.02 por 100K borrados

Las ineficiencias actuales con costo real son:

**1. Doble lectura en `update()` y `delete()`**

`FireStoreAdapter.update()` y `FireStoreAdapter.delete()` hacen `docRef.get()` para verificar existencia antes de la operación, y luego `update()` hace otro `docRef.get()` para retornar el documento actualizado:

```
update(id, data):
  1. docRef.get()           → 1 lectura (existencia)
  2. docRef.update(data)    → 1 escritura
  3. docRef.get()           → 1 lectura (retorno del doc actualizado)
  Total: 2 lecturas + 1 escritura por PATCH
```

El `GroupService.update()` agrava esto: para un diff de N usuarios, ejecuta N veces `userService.update()`, cada una costando 2 lecturas + 1 escritura. Un PATCH de grupo con 10 usuarios en el diff = 20 lecturas + 10 escrituras solo para sync, más las del grupo mismo.

**2. `RedirectServiceApi.create()` hace una lectura extra para verificar unicidad de path**

```
create(redirect):
  1. getByPath(path)   → query (1 lectura mínima)  — verificación de unicidad
  2. db.create(data)   → 1 escritura
  3. docRef.get()      → 1 lectura (retorno del doc creado)
  Total: 2 lecturas + 1 escritura por POST /redirects
```

Lo mismo aplica a `UserServices.create()` y `GroupService.create()`.

**3. `FireStoreAdapter` instancia un cliente Firestore por colección**

Cada instancia crea `new Firestore.Firestore()`. Con tres colecciones activas (redirects, users, groups), hay tres instancias del cliente. El SDK maneja connection pooling internamente pero es un patrón que **impide batch writes y transactions entre colecciones** — crítico para el sync atómico de `GroupService.update()`.

**4. Queries de lista sin índice compuesto documentado**

El `GET /api/v1/redirects` para usuarios no-admin usa `Filter.or(owner==email, permission array-contains-any [...])`. Esta query requiere un índice compuesto en Firestore. Si el índice no existe, la query falla en producción con `FAILED_PRECONDITION`. Los índices no están en `firestore.indexes.json`.

### §1.1 — Singleton del cliente Firestore

**Qué cambia:** extraer `new Firestore.Firestore()` de `FireStoreAdapter` a un módulo singleton compartido.

**Archivo nuevo:** `src/lib/firestore-client.js`

```js
// src/lib/firestore-client.js
const { Firestore } = require('@google-cloud/firestore');
const firestoreClient = new Firestore();
module.exports = firestoreClient;
```

**Cambio en `FireStoreAdapter`:**

```js
// src/lib/firestore.js — constructor antes:
this.db = new Firestore.Firestore();

// constructor después:
const firestoreClient = require('./firestore-client');
this.db = firestoreClient;
```

**Impacto:** no-breaking. El API pública de `FireStoreAdapter` no cambia. Permite batch writes en §1.3.

**Costo eliminado:** cero (el SDK ya comparte el pool). El valor es arquitectónico: habilita §1.3.

### §1.2 — Eliminar la doble lectura en update()

**Problema:** `update()` hace `get()` para verificar existencia y luego otro `get()` para retornar el documento. Esto es 2 lecturas + 1 escritura en lugar de 1 lectura + 1 escritura (si se retorna solo el campo `updated` o si se confía en la semántica de `update()`).

**Decisión de diseño:** Firestore `.update()` lanza `NOT_FOUND` si el documento no existe — no necesita la lectura previa de existencia. El segundo `get()` post-update puede reemplazarse por reconstruir el objeto con los datos conocidos.

Sin embargo, retornar el documento actualizado desde Firestore garantiza consistencia si otro proceso modificó el documento concurrentemente. Para este sistema de baja concurrencia, la eliminación de la segunda lectura es aceptable.

**Cambio en `FireStoreAdapter.update()`:**

```js
async update(id, data) {
  const docRef = this.collection.doc(id);
  try {
    await docRef.update({
      updated: Firestore.Timestamp.fromMillis(Date.now()),
      ...data,
    });
  } catch (err) {
    if (err.code === 5) { // gRPC NOT_FOUND
      throw boom.notFound('Resource not found');
    }
    throw err;
  }
  return await docRef.get(); // mantener esta lectura — necesaria para el docParser
}
```

Nota: se elimina la lectura de existencia previa (`if (!(await docRef.get()).exists)`), confiando en el error gRPC `NOT_FOUND` (código 5) que lanza Firestore cuando el documento no existe.

**Costo eliminado:** 1 lectura por cada `PATCH` en redirects, users, y groups.

**Archivos:** `src/lib/firestore.js`, `src/lib/__test__/firestore.test.js`

### §1.3 — Sync atómico de Group.users ↔ User.groups con batch write

**Problema:** `GroupService.update()` ejecuta N llamadas `userService.update()` en un bucle secuencial. Si falla a mitad del loop, el estado queda inconsistente: algunos usuarios actualizados, otros no. Además, cada iteración del loop genera 2 lecturas + 1 escritura.

**Solución:** Firestore `WriteBatch`. Un batch agrupa hasta 500 operaciones que se aplican atómicamente: o todas o ninguna.

**Cambio en `GroupService.update()`:**

En lugar del loop secuencial de `userService.update()`, construir un `WriteBatch` con todas las operaciones:

```js
// Esquema del cambio (no código final):
const batch = firestoreClient.batch();

for (const email of added) {
  const user = userMap.get(email);
  const userRef = firestoreClient.collection('users').doc(user.id);
  batch.update(userRef, { groups: [...user.groups, current.slug], updated: now });
}

for (const email of removed) {
  const user = userMap.get(email);
  const userRef = firestoreClient.collection('users').doc(user.id);
  batch.update(userRef, { groups: user.groups.filter(g => g !== current.slug), updated: now });
}

const groupRef = firestoreClient.collection('groups').doc(id);
batch.update(groupRef, { ...updateGroupParser(group), updated: now });

await batch.commit(); // atómico
```

**Prerequisito:** §1.1 (singleton del cliente Firestore).

**Costo eliminado:** el loop de N usuarios pasa de `N * (2 lecturas + 1 escritura)` a `N * 0 lecturas + N * 1 escritura batch` (batch writes siguen cobrándose por documento, pero se elimina la doble lectura por usuario en `update()`). El fetch-first (`userMap`) sigue siendo necesario para el diff, pero se hace una sola vez antes del batch.

**Costo real del batch:** Firestore cobra las operaciones del batch individualmente. La ganancia es N lecturas eliminadas (la verificación de existencia en `update()`), no N escrituras.

**Archivos:** `src/api/groups/services/group.service.api.js`, `src/lib/firestore-client.js`

### §1.4 — firestore.indexes.json versionado y script de gestión

**Problema:** los índices compuestos no están sincronizados de forma reproducible. Si se pierde el proyecto GCP o se crea un entorno nuevo, la API falla silenciosamente con `FAILED_PRECONDITION`.

**Decisiones de diseño:**

- El proyecto usa `gcloud` exclusivamente (no Firebase CLI). No se crea `firebase.json`.
- La gestión de índices se realiza mediante `scripts/manage-indexes.js` en la raíz del repositorio, que usa `gcloud` como runtime de CLI.

**Índices compuestos requeridos actualmente:**

```json
{
  "indexes": [
    {
      "collectionGroup": "redirects",
      "queryScope": "COLLECTION",
      "fields": [
        { "fieldPath": "owner", "order": "ASCENDING" },
        { "fieldPath": "updated", "order": "DESCENDING" }
      ]
    },
    {
      "collectionGroup": "redirects",
      "queryScope": "COLLECTION",
      "fields": [
        { "fieldPath": "permission", "arrayConfig": "CONTAINS" },
        { "fieldPath": "updated", "order": "DESCENDING" }
      ]
    }
  ],
  "fieldOverrides": []
}
```

El índice para `Filter.or(owner==, permission array-contains-any)` requiere que Firestore pueda ordenar por `updated` en ambas ramas del OR. Firestore evalúa cada rama del OR por separado y hace un merge, así que se necesitan los dos índices anteriores.

**Batching de `array-contains-any` — descoped de v4:**

El operador `array-contains-any` acepta un máximo de 10 valores. La guardia `.max(10)` en §0.1 garantiza que ningún usuario pueda superar ese límite, por lo que el handler existente con `Filter.or()` es la solución definitiva para v4.

El método `findByPermissions(email, groups, options)` con soporte de batching **no se implementa en v4**. Si el límite de §0.1 se eleva en el futuro, implementarlo siguiendo el runbook `docs/runbooks/increase-group-limit.md`.

**Script `scripts/manage-indexes.js` — comportamiento especificado:**

El script sincroniza los índices del repositorio contra el proyecto GCP activo:

1. Lee `firestore.indexes.json` de la raíz del repositorio como fuente de verdad.
2. Ejecuta `gcloud firestore indexes composite list --format=json` para obtener los índices actuales del proyecto.
3. Compara los índices deseados (del JSON) contra los existentes (de gcloud). La comparación se hace por `collectionGroup + queryScope + fields`, ignorando los campos `__name__` (añadido automáticamente por Firestore), `density` (campo interno de Firestore ausente en `firestore.indexes.json`), `state`, y el `name` opaco (ID asignado por GCP).
4. **Crea** los índices del JSON que no existen en el proyecto.
5. **Elimina** los índices existentes que no están en el JSON (índices obsoletos). La eliminación requiere confirmación interactiva; con el flag `--force` la omite (para uso en CI/CD).
6. Trata índices en estado `CREATING` como ya existentes — no los duplica.
7. Lee el proyecto GCP de `gcloud config get-value project` — no está hardcodeado en el script.
8. Es idempotente: ejecutarlo dos veces sin cambios no produce ninguna operación.

**Comandos npm:**

- `npm run indexes`: `node scripts/manage-indexes.js`
- `npm run deploy`: ver ambigüedad P6 más abajo.

**Archivos nuevos:** `scripts/manage-indexes.js`

**`firebase.json`:** no se crea. Descartado.

## §2 — Enforcement de permisos en redirects

### Estado actual

El campo `permission: string[]` en `Redirect` acepta entradas con formato `"read:{group}"`, `"edit:{group}"`, `"delete:{group}"` — validado en el schema Joi. Sin embargo:

- Solo `"read:{group}"` tiene enforcement: el filtro `array-contains-any` en `GET /` y la verificación inline en `GET /:id`.
- `"edit:{group}"` y `"delete:{group}"` se almacenan en Firestore pero **ningún handler los verifica**.
- `PATCH /:id` y `DELETE /:id` solo permiten owner o admin.

### Visión del usuario para v4

El usuario quiere que el propietario de un redirect pueda configurar:
1. Visibilidad: solo él, o el grupo al que pertenece el redirect.
2. Si el grupo puede verlo: si además puede editarlo y/o eliminarlo.

### Decisiones de diseño adoptadas

**D-§2-1 — El modelo de permisos existente (`permission: string[]`) ya soporta esta visión. No se cambia el schema.**

El array ya admite `"read:{group}"`, `"edit:{group}"`, `"delete:{group}"`. Lo que falta es solo el enforcement en los handlers.

**D-§2-2 — "El grupo al que pertenece el redirect" se define por el namespace del path, no por un campo explícito.**

Un redirect con path `/fc/seminar` pertenece al grupo `fc`. No se añade un campo `group` al modelo `Redirect` — se infiere del path. Esta inferencia es posible porque el namespace enforcement en `POST /` garantiza que el primer segmento es siempre un slug de grupo válido para usuarios no-admin.

Excepción: paths de admin (`/promo`, `/internal/...`) no tienen grupo inferible. Los permisos de esos redirects solo aplican si el admin los añade explícitamente al array `permission`.

**D-§2-3 — La interfaz de permisos del cliente se simplifica a tres opciones.**

El cliente (futuro frontend) puede enviar en `permission`:
- `[]` o ausente: solo el owner puede ver, editar y eliminar.
- `["read:{group}"]`: el grupo puede ver. Solo el owner puede editar y eliminar.
- `["read:{group}", "edit:{group}"]`: el grupo puede ver y editar. Solo el owner puede eliminar.
- `["read:{group}", "edit:{group}", "delete:{group}"]`: el grupo puede ver, editar y eliminar (excepto el owner siempre puede todo).

El schema Joi ya valida este formato. No requiere cambios.

### §2.1 — Enforcement de `edit:{group}` en PATCH /:id

**Qué cambia:** el handler `PATCH /api/v1/redirects/:id` actualmente solo permite al owner o admin. Se amplía para permitir también a usuarios que pertenezcan a un grupo con `"edit:{group}"` en el array `permission` del redirect.

**Lógica de acceso:**

```js
const editPermissions = req.user.groups.map(g => `edit:${g}`);
const canEdit =
  req.user.role === 'admin' ||
  existing.owner === req.user.email ||
  (existing.permission || []).some(p => editPermissions.includes(p));
if (!canEdit) return next(boom.forbidden('Insufficient permissions'));
```

**Archivo:** `src/api/redirect/routes/redirect.route.api.js`

### §2.2 — Enforcement de `delete:{group}` en DELETE /:id

**Qué cambia:** análogo a §2.1 para el handler DELETE.

**Lógica de acceso:**

```js
const deletePermissions = req.user.groups.map(g => `delete:${g}`);
const canDelete =
  req.user.role === 'admin' ||
  existing.owner === req.user.email ||
  (existing.permission || []).some(p => deletePermissions.includes(p));
if (!canDelete) return next(boom.forbidden('Insufficient permissions'));
```

**Archivo:** `src/api/redirect/routes/redirect.route.api.js`

### §2.3 — Verificación de existencia del grupo en POST /:id

**Problema actual (deuda técnica 3.4 de la auditoría):** `POST /api/v1/redirects` verifica que el usuario pertenece al grupo por el JWT, pero no verifica que el `Group` documento existe en Firestore. Un usuario podría crear paths bajo un grupo eliminado.

**Qué cambia:** antes de construir el `fullPath`, verificar que el slug del grupo existe en la colección `groups` usando `GroupService.getBySlug(group)`.

**Prerequisito de inyección:** el handler de redirects necesita acceso a `GroupService`. Opciones:
- Instanciar `GroupService` dentro del router (acoplamiento, pero consistente con el patrón actual donde cada router instancia su service).
- Pasar `GroupService` como dependencia al router (más limpio, requiere refactor del mount point).

**Decisión adoptada:** instanciar `GroupService` dentro del redirect router, igual que el patrón actual. El router es el composition root para sus dependencias directas.

**Archivo:** `src/api/redirect/routes/redirect.route.api.js`

### §2.4 — Tests de los nuevos accesos de permisos

Tests necesarios:
- `PATCH /:id` permite a usuario con `edit:{group}` en el permission array.
- `PATCH /:id` rechaza a usuario sin `edit:{group}`.
- `DELETE /:id` permite a usuario con `delete:{group}`.
- `DELETE /:id` rechaza a usuario sin `delete:{group}`.
- `POST /` rechaza cuando el grupo no existe en Firestore.

**Archivo:** `src/api/redirect/routes/__test__/redirect.route.api.test.js`

## §3 — API Keys de usuario

### Estado actual

El modelo `User` tiene un campo `auth.apiToken: string` en Firestore. No existe ningún endpoint para generar, listar, revocar o usar API Keys. El campo es un string único, no una colección de keys con metadatos.

### Visión del usuario para v4

Los usuarios pueden crear API Keys con:
- **nombre:** descripción de uso (ej: "CI pipeline", "script personal")
- **vigencia:** fecha de expiración o null para no-expirar
- **scopes:** subconjunto explícito de operaciones permitidas — nunca se heredan automáticamente

### Modelo de scopes (P1 — RESUELTA)

**Decisión confirmada por el usuario:** las API Keys deben tener scopes granulares para limitar el blast radius en caso de filtración. Un admin no crea una key "con todos los poderes de admin" — crea una key con los permisos mínimos necesarios para ese caso de uso específico.

**Catálogo de scopes disponibles:**

| Scope | Operaciones permitidas | Disponible para |
|-------|----------------------|-----------------|
| `read:redirects` | `GET /api/v1/redirects`, `GET /api/v1/redirects/:id` | todos |
| `write:redirects` | `POST`, `PATCH`, `DELETE /api/v1/redirects` (+ read implícito) | todos |
| `read:users` | `GET /api/v1/users`, `GET /api/v1/users/:id`, `GET /api/v1/users/me` | solo admin |
| `write:users` | `POST`, `PATCH`, `DELETE /api/v1/users` (+ read:users implícito) | solo admin |
| `read:groups` | `GET /api/v1/groups`, `GET /api/v1/groups/:id` | solo admin |
| `write:groups` | `POST`, `PATCH`, `DELETE /api/v1/groups` (+ read:groups implícito) | solo admin |

**Reglas de combinación:**
- Una key puede tener múltiples scopes: `["read:redirects", "read:users"]`.
- Los scopes `read:users`, `write:users`, `read:groups`, `write:groups` solo pueden asignarse a keys de usuarios con `role === 'admin'`. El handler de `POST /api/v1/users/me/api-keys` rechaza con 403 si un usuario regular intenta crear una key con scopes de admin.
- Un scope `write:*` no implica automáticamente el scope `read:*` correspondiente en la representación almacenada — el enforcement los verifica por separado. Sin embargo, en la práctica los handlers de escritura también leen el recurso primero (fetch-first en update/delete), por lo que tener `write:redirects` sin `read:redirects` no es un caso de uso útil. El frontend debería preseleccionar `read:redirects` al seleccionar `write:redirects`.
- No existe un scope `admin:full`. Si un admin necesita acceso completo, crea una key con todos los scopes relevantes de forma explícita.

**Schema actualizado del campo `scope` en el documento `apiKey`:**

```
scope: string[]   // subconjunto del catálogo; al menos un scope requerido
                  // ej: ["read:redirects"]
                  // ej: ["read:redirects", "write:redirects", "read:users"]
```

**Enforcement en el middleware:**

El middleware `authenticate` construye `req.user.apiKey.scopes` (array). El nuevo middleware `authorizeApiKeyScope(requiredScope)` verifica que el array lo contenga:

```js
// src/middleware/authorize-api-key-scope.middleware.js
function authorizeApiKeyScope(requiredScope) {
  return (req, next) => {
    if (!req.user.apiKey) return next(); // autenticado con JWT — no aplica
    if (!req.user.apiKey.scopes.includes(requiredScope)) {
      return next(boom.forbidden(`API Key scope '${requiredScope}' required`));
    }
    next();
  };
}
```

Este middleware se encadena después de `authenticate` en cada router que acepte API Keys:

```js
// Solo redirects aceptan API Keys
redirectRouterApi.use(authenticate);
redirectRouterApi.get('/',    authorizeApiKeyScope('read:redirects'),  handler);
redirectRouterApi.get('/:id', authorizeApiKeyScope('read:redirects'),  handler);
redirectRouterApi.post('/',   authorizeApiKeyScope('write:redirects'), handler);
redirectRouterApi.patch('/:id',  authorizeApiKeyScope('write:redirects'), handler);
redirectRouterApi.delete('/:id', authorizeApiKeyScope('write:redirects'), handler);
```

Los routers de users y groups rechazan API Keys completamente — verifican que `req.user.apiKey === undefined` al inicio (o que el rol sea admin con scope adecuado si se decide habilitar en el futuro).

### §3.1 — Modelo de datos: subcolección `apiKeys`

**Decisión de diseño:** las API Keys se almacenan como subcolección `users/{userId}/apiKeys/{keyId}`, no como array en el documento de usuario.

**Razón:** un array de objetos en el documento de usuario tiene límite de 1MB por documento y no permite queries individuales por key. Una subcolección permite queries, tiene su propio control de acceso Firestore, y no contamina el documento principal del usuario.

**Schema del documento `apiKey`:**

```
users/{userId}/apiKeys/{keyId}:
  name:       string        // descripción del uso
  keyHash:    string        // SHA-256 del token; el token plano solo se muestra una vez al crear
  prefix:     string        // primeros 8 chars del token para identificación (ej: "sk_1kg_ab")
  scopes:     string[]      // subconjunto del catálogo de scopes; ej: ["read:redirects", "write:redirects"]
  expiresAt:  Timestamp | null  // null = no expira
  createdAt:  Timestamp
  lastUsedAt: Timestamp | null  // null si nunca usada
  active:     boolean       // false = revocada
```

**El token plano nunca se almacena.** Solo se retorna al cliente en el momento de creación. El servidor almacena solo el hash SHA-256.

**Formato del token:** `sk_1kg_{base62_random_32_chars}`. El prefijo `sk_1kg_` permite identificar el sistema de origen en logs y es reconocible para el usuario.

### §3.2 — Autenticación con API Key

**Decisión de diseño:** las API Keys se aceptan en el header `Authorization: Bearer sk_1kg_...`. El middleware `authenticate` las distingue del JWT por el prefijo `sk_1kg_`.

**Alternativa rechazada:** header separado `X-API-Key`. El estándar Bearer es más compatible con herramientas existentes (curl, Postman, swagger-ui).

**Flujo de autenticación con API Key:**

1. Request llega con `Authorization: Bearer sk_1kg_abc...`
2. `authenticate` detecta el prefijo `sk_1kg_`
3. Calcula `SHA-256(token)` → `keyHash`
4. Query: `collectionGroup('apiKeys').where('keyHash', '==', keyHash).limit(1)`
5. Verifica: `active === true`, `expiresAt === null || expiresAt > now`
6. Obtiene el `userId` del path del documento (el padre de la subcolección)
7. Carga el usuario: `users.findOne(userId)`
8. Construye `req.user` con el scope de la key en lugar del rol completo del usuario

**Índice requerido:** `collectionGroup('apiKeys')` con `keyHash` — índice de colección en modo `Collection Group`.

**Performance:** esta query es una lectura por request autenticado con API Key. Es inevitable — el hash no puede decodificarse a userId sin la lookup. **El caché en memoria es requerido, no opcional:** la instancia singleton de `node-cache` (ya existente en `src/utils/cache.js`) debe cachear el resultado del lookup usando el `keyHash` como clave, con TTL de 30 segundos. Sin este caché, un script que haga 100 requests en 30 segundos genera 100 lecturas Firestore en lugar de 1. La revocación de una key (`active = false`) surte efecto en un máximo de 30 segundos tras el cambio — esto es aceptable y debe documentarse en la respuesta del `DELETE /api/v1/users/me/api-keys/:keyId`.

### §3.3 — Endpoints CRUD de API Keys

Montados en `/api/v1/users/me/api-keys` — siempre bajo el usuario autenticado. No existe endpoint de admin para listar keys de otro usuario (privacidad por diseño).

```
GET    /api/v1/users/me/api-keys           Lista las keys del usuario autenticado (sin el token plano)
POST   /api/v1/users/me/api-keys           Crea una nueva key (retorna el token plano UNA sola vez)
DELETE /api/v1/users/me/api-keys/:keyId    Revoca una key (active = false, no borra el doc)
```

No existe `PATCH` — las keys son inmutables una vez creadas. Para cambiar scope o expiración, revocar y crear nueva.

**Request body de POST:**

```json
{
  "name": "CI pipeline produccion",
  "scopes": ["read:redirects"],
  "expiresAt": "2027-01-01T00:00:00Z"
}
```

`expiresAt` es opcional — si se omite o es `null`, la key no expira.

**Response de POST (única vez que se muestra el token):**

```json
{
  "message": "api key created",
  "data": {
    "id": "abc123",
    "name": "CI pipeline produccion",
    "prefix": "sk_1kg_ab",
    "scopes": ["read:redirects"],
    "expiresAt": "2027-01-01T00:00:00Z",
    "createdAt": "2026-06-10T12:00:00Z",
    "token": "sk_1kg_abcdefghij..."
  }
}
```

**Response de GET (sin token):**

```json
{
  "message": "api keys retrieved",
  "data": [
    {
      "id": "abc123",
      "name": "CI pipeline produccion",
      "prefix": "sk_1kg_ab",
      "scopes": ["read:redirects"],
      "expiresAt": "2027-01-01T00:00:00Z",
      "createdAt": "2026-06-10T12:00:00Z",
      "lastUsedAt": null,
      "active": true
    }
  ]
}
```

### §3.4 — Enforcement de scope en endpoints de redirects

Cuando el request se autentica con API Key (no JWT), solo los endpoints de redirects son accesibles. Los routers de users y groups verifican `req.user.apiKey === undefined` y rechazan con 403 si viene una API Key (con excepción de los scopes de admin cuando el usuario tiene `role === 'admin'` y su key incluye `read:users` / `write:users` / `read:groups` / `write:groups`).

El middleware `authorizeApiKeyScope(requiredScope)` se encadena en cada ruta. Ver la sección "Modelo de scopes" para la implementación completa.

**Cambio en `req.user` cuando se autentica con API Key:**

```js
req.user = {
  userId: user.id,
  email: user.email,
  role: user.role,
  groups: user.groups,
  apiKey: {
    id: keyDoc.id,
    scopes: keyDoc.scopes,  // string[] — subconjunto del catálogo
  }
}
```

Los handlers existentes no necesitan cambios — `req.user.email`, `req.user.role`, `req.user.groups` funcionan igual. El campo `apiKey.scopes` solo es leído por `authorizeApiKeyScope`.

}

## Preguntas abiertas

### P1 — RESUELTA

**Respuesta del usuario:** las API Keys deben tener scopes granulares por recurso/operación para limitar el blast radius en caso de filtración. Un admin no crea una key con todos los poderes de admin — elige el subconjunto mínimo necesario.

**Decisión adoptada:** catálogo de 6 scopes (`read:redirects`, `write:redirects`, `read:users`, `write:users`, `read:groups`, `write:groups`). Campo `scopes: string[]` en el documento apiKey. No existe `admin:full`. Ver §3 "Modelo de scopes" para el detalle completo. §3.1, §3.3, §3.4, §3.5 actualizados en consecuencia.

### P2 — RESUELTA por defecto razonable

**Decisión adoptada:** las API Keys heredan exactamente las mismas reglas de acceso que la sesión JWT del usuario. El scope `read:redirects` solo bloquea operaciones de escritura — no restringe qué redirects puede leer. Un usuario con `"read:fc"` en su permission sigue viendo esos redirects al usar una API Key con `read:redirects`.

**Razón:** restringir a "solo redirects propios" crearía un sistema de permisos paralelo inconsistente con el modelo de grupos ya implementado. El valor de los scopes es limitar operaciones (leer vs. escribir), no reemplazar el modelo de visibilidad.

**Si el usuario quiere cambiar esto:** abrir una issue antes de Sprint 4.

### P3 — RESUELTA por defecto razonable

**Decisión adoptada:** eliminar `auth.apiToken` del modelo `User` y del schema Joi al implementar §3. No se realiza migración.

**Razón:** la rama `main` (producción) no tiene el endpoint de API activo — no hay usuarios con `auth.apiToken` populado desde código en producción. La rama `dev` es desarrollo activo. Si existe algún documento en Firestore con el campo, Firestore simplemente lo ignora cuando el modelo deja de leerlo (no produce error). El campo muerto en Firestore no contamina las queries ni el billing.

**Acción en §3.5:** el constructor de `User` y su schema Joi eliminan la referencia a `auth.apiToken`.

### P4 — RESUELTA por defecto razonable

**Decisión adoptada:** hacer §4.2 al principio de v4, antes de cualquier bloque funcional, en la rama `dev` limpia.

**Razón:** hacerlo al principio minimiza conflictos de merge. Una vez que los 10 archivos están renombrados y todos sus imports actualizados en un solo commit `[refactor]`, el resto del trabajo de v4 usa ya los nombres nuevos sin deuda pendiente. Diferirlo a v5 significa que todo el código de v4 sigue usando nombres inconsistentes que hay que renombrar después.

**Orden de ejecución revisado:** §4.1 → §4.2 → BUG-A → BUG-D → §1.x → §2.x → §3.x. Ver plan de implementación actualizado al final.

### P5 — RESUELTA por defecto razonable

**Decisión adoptada:** máximo 10 API Keys activas por usuario, validado en el handler de `POST /api/v1/users/me/api-keys`.

**Razón:** el sistema es de baja escala pero un límite de 10 es más que suficiente para los casos de uso previstos (CI pipelines, scripts personales). Sin límite, un bug o un loop en un script podría crear miles de keys silenciosamente. El límite se cuenta sobre keys con `active === true` — las revocadas no cuentan.

**Implementación:** `GET count(*) where active == true` en la subcolección antes de crear. Si `count >= 10`, responder 422 con `"Maximum of 10 active API Keys per user"`.

**Si el usuario quiere cambiar el límite:** añadir `MAX_API_KEYS_PER_USER` a `src/config/index.js` con valor por defecto `10`.

### P6 — RESUELTA

**Decisión:** `npm run indexes` y `npm run deploy` son comandos separados e independientes. El script `deploy` no cambia — `gcloud app deploy` sin argumentos detecta `app.yaml` en el directorio raíz automáticamente.

**Razón:** el desarrollador (o el pipeline de CI) los corre explícitamente en orden. Un fallo de sincronización de índices no bloquea el deploy de la aplicación cuando no hay cambios de índices.

**Acción en §1.4:** `package.json` añade `"indexes": "node scripts/manage-indexes.js"`. El script `deploy` permanece igual (`gcloud app deploy`).
