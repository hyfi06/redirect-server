# Spec v4.1.3 — Rate limiting en redirect y script de logs

**Fecha:** 2026-06-27
**Rama base:** dev (v4.1.2 completa)
**Estado al iniciar v4.1.3:** API v1 completa, autenticada, suite E2E passing. Todos los ítems de v4.1.1 (limpieza de código muerto, DRY) cerrados en la rama dev.

---

## Contexto y motivación

v4.1.3 es un sprint de hardening operacional. Dos objetivos:

1. **Rate limiting en el catch-all de redirect:** La superficie pública `GET /*` es accesible sin autenticación. Sin límite de tasa, una IP puede agotar la cuota de Firestore o disparar costos de instancia con tráfico sintético. Se aplica un límite in-memory per-IP con `express-rate-limit`.

2. **Script de logs de producción:** El diagnóstico de errores en producción requiere acceder a Cloud Logging. El script `npm run logs` encapsula el comando `gcloud logging read` con filtros y formato legible, reduciendo la fricción operacional.

---

## §1 — Rate limiting en el catch-all de redirect

### Motivación y decisión

El catch-all `GET /*` es la única ruta pública sin autenticación del servicio. Las rutas `/api/v1/**` están protegidas por `authenticate` — un cliente no autenticado recibe 401 antes de llegar a Firestore. El catch-all no tiene esa protección: cualquier IP puede ejecutar N lecturas de Firestore por minuto sin restricción.

Con GAE escalando a 0–3 instancias y la cuota de Firestore cobrada por lectura, un abuso sostenido podría generar costos inesperados incluso a tráfico bajo.

**Decisión:** aplicar `express-rate-limit` (in-memory store) en el catch-all de redirect, con ventana de 1 minuto y límite de 60 requests/min/IP.

### Trade-offs aceptados

**Multi-instancia (3× bypass):** Con hasta 3 instancias de GAE, cada instancia mantiene su propio `MemoryStore`. Una IP puede distribuir requests entre instancias y alcanzar hasta 3× el límite configurado antes de ser bloqueada en alguna instancia. Este trade-off es aceptado — la alternativa (Redis/Memcache compartido) añade infraestructura desproporcionada al riesgo actual.

**No se aplica a `/api/v1/**`:** Las rutas de API ya están detrás de `authenticate`. Un cliente no autenticado jamás llega al handler. Rate limiting en esa superficie agregaría complejidad sin beneficio proporcional en el escenario de uso actual.

### Parámetros de configuración

| Parámetro | Valor sugerido | Notas |
|-----------|---------------|-------|
| Ventana | 60 segundos | Granularidad mínima razonable para uso legítimo |
| Límite | 60 requests/ventana/IP | 1 req/s promedio — permite burst corto sin bloquear uso normal |
| Respuesta | 429 JSON | Consistente con el formato Boom del proyecto |

El backend-engineer puede ajustar estos parámetros. Se documentan aquí como punto de partida razonado, no como constraint inamovible.

### Respuesta en límite excedido

La respuesta 429 debe seguir el formato Boom del proyecto:

```
HTTP 429
{ "statusCode": 429, "error": "Too Many Requests", "message": "..." }
```

`express-rate-limit` permite configurar un handler `handler(req, res)` personalizado para producir ese JSON exacto. No se usa la respuesta por defecto de la librería.

### Ubicación del middleware

El rate limiter debe aplicarse **antes** del lookup de caché/Firestore en el catch-all. Con el flujo actual:

```
GET /*
  ↓
[rate limiter]        ← nuevo, primer middleware del router
  ↓
nodeCache.has(path)?
  ├── HIT  → redirect
  └── MISS → Firestore → cache → redirect
```

Esto evita lecturas de Firestore y de caché en requests rechazados por rate limit.

### Archivos afectados

| Archivo | Cambio |
|---------|--------|
| `package.json` | Nueva dependencia `express-rate-limit` |
| `src/redirect/routes/redirect.router.js` | Aplicar el middleware de rate limiting |

## §2 — Script de logs de producción

### Motivación y decisión

El diagnóstico de errores en producción requiere leer Cloud Logging. Sin un comando específico, el desarrollador debe recordar la sintaxis completa de `gcloud logging read` con filtros de proyecto, severidad, y número de entradas — o navegar la consola GCP.

**Decisión:** añadir un script Node.js `scripts/logs.js` que encapsule `gcloud logging read` con los filtros correctos y formatee la salida para lectura en terminal.

Dos comandos npm:

| Comando | Comportamiento |
|---------|---------------|
| `npm run logs` | Últimos 100 logs de severidad ERROR o superior |
| `npm run logs:all` | Últimos 100 logs de la app (todas las severidades) |

### Diseño del script

**Auto-detección del proyecto:** El script invoca `gcloud config get-value project` para obtener el project ID activo. Sin hardcodeo — el desarrollador puede cambiar de proyecto GCP con `gcloud config set project` sin modificar el script.

**Sin dependencias npm nuevas:** El script usa solo `child_process` de Node.js estándar para invocar `gcloud`. No se añaden dependencias al proyecto por código que solo se ejecuta en desarrollo local.

**Distinción entre modos:** El script detecta si fue invocado con el flag `--all` para decidir el filtro de severidad. Los dos scripts de npm pasan ese flag según corresponde.

**Formato de salida:** Cada entrada de log se imprime en una línea legible en terminal:

```
[YYYY-MM-DD HH:MM:SS] [SEVERITY] message
```

No JSON crudo — la salida de `gcloud logging read --format=json` se parsea en el script y se reformatea. Si el campo `message` no está disponible, se usa `textPayload` o `jsonPayload` como fallback.

**Filtros de `gcloud logging read`:**

| Filtro | `npm run logs` | `npm run logs:all` |
|--------|---------------|-------------------|
| `resource.type` | `"gae_app"` | `"gae_app"` |
| `severity` | `>= ERROR` | sin filtro |
| `--limit` | 100 | 100 |
| `--project` | auto-detectado | auto-detectado |
