**Fecha:** 2026-06-25
**Rama base:** dev (v4.0.4)
**Estado al iniciar v4.1:** API v1 completa, autenticada, 100% tests passing. API Keys, permisos de redirect, batch writes e índices Firestore implementados y cerrados en v4. Release de maduración: sin features nuevas para el usuario — mejoras operacionales, deuda técnica e infraestructura de tests.

## Contexto y motivación

El panel de App Engine muestra que más del 80% de las solicitudes son probes automáticas de bots que escanean vulnerabilidades conocidas (PHP, WordPress, dotfiles). Cada probe llega al catch-all de redirect, ejecuta un query en Firestore y devuelve 404 — consumiendo lecturas de Firestore y CPU innecesariamente.

Datos reales de las últimas 24h del entorno de producción:

| Path | Solicitudes | Naturaleza |
|------|-------------|------------|
| `/` | 45 | Legítimo (página de inicio) |
| `/robots.txt` | 13 | Legítimo (crawler) |
| `/wp-login.php`, `/.env`, `/admin.php`, etc. | 3–6 cada una | Bot probe (latencia 50–100ms: llegan a Firestore) |

El impacto es doble:

1. **Costo innecesario:** cada probe genera al menos 1 lectura Firestore ($0.06/100K lecturas).
2. **Capacidad desperdiciada:** el ciclo completo request → Firestore query → 404 para paths que nunca existirán.

## §1 — Middleware de rechazo de bots

### Problema

El orden de montaje actual en `src/app.js` es:

```
rootRouter     →  GET /         Static HTML + assets públicos
apiV1          →  /api/v1/**    REST API
redirectRoute  →  GET /*        Catch-all: URL shortener
```

Todo path no resuelto por `rootRouter` o `apiV1` llega a `redirectRoute`. Este router consulta Firestore y responde 404 si el path no existe. Los bots generan tráfico artificial sobre este camino.

### Solución

Agregar un middleware de rechazo temprano entre `apiV1` y `redirectRoute` que intercepte paths con patrones conocidos de scanners antes de que lleguen a Firestore.

### Patrones a rechazar

| Categoría | Descripción del patrón | Ejemplos representativos |
|-----------|------------------------|--------------------------|
| Scripts PHP | Path termina con extensión `.php` | `/wp-login.php`, `/admin.php`, `/file.php` |
| Dotfiles y directorios ocultos | Path empieza con `/.` | `/.env`, `/.git`, `/.htaccess` |
| WordPress | Primer segmento del path es `/wp-` | `/wp-admin/`, `/wp-includes/`, `/wp-content/` |
| Scripts de servidor (otros) | Path termina con `.asp`, `.aspx`, `.jsp`, o `.cgi` | `/admin.asp`, `/index.jsp`, `/cgi-bin/test.cgi` |

El matching de extensiones y prefijos debe ser **case-insensitive** para cubrir variantes como `.PHP`, `/WP-admin/`.

### Comportamiento especificado

- Responde HTTP 404. No se responde 403 para no revelar al scanner si la distinción "existe / no existe" aplica.
- No consulta Firestore.
- No interactúa con `node-cache`.
- No registra logs de error — estos requests son ruido esperado y no representan errores de la aplicación.
- Los paths de la API (`/api/v1/...`) ya son resueltos por `apiV1` antes de llegar al middleware; el middleware no los intercepta ni necesita excluirlos explícitamente.

### Archivos afectados

| Archivo | Cambio |
|---------|--------|
| `src/middleware/bot-reject.middleware.js` | Nuevo — lógica del middleware |
| `src/app.js` | Modificado — montaje entre `apiV1` y `redirectRoute` |
| `src/middleware/__test__/bot-reject.middleware.test.js` | Nuevo — tests |

## §2 — robots.txt mejorado

### Estado actual

`src/public/robots.txt` contiene únicamente directivas de WordPress:

```
User-agent: *
Disallow: /wp-admin/
Disallow: /wp-includes/
Disallow: /*/wp-includes/
Disallow: /*/wp-content/
```

Las directivas disuaden a crawlers obedientes de indexar rutas de WordPress que no existen en este servicio. Sin embargo, no cubren la API REST del proyecto ni declaran explícitamente qué está permitido indexar.

### Solución

Actualizar `src/public/robots.txt` para:

1. Declarar `Allow: /` explícitamente — la página pública del servicio es indexable.
2. Prohibir la indexación de la API REST con `Disallow: /api/`.
3. Conservar las directivas existentes de WordPress (siguen siendo útiles para crawlers obedientes).

### Efecto esperado

Un crawler obediente (Googlebot, Bingbot) que lea el `robots.txt` actualizado:
- Indexará la página de inicio `/`.
- No intentará rastrear rutas de la API.
- No intentará indexar rutas de WordPress.

Los bots maliciosos que ignoran `robots.txt` seguirán siendo bloqueados por §1 antes de llegar a Firestore.

**Nota:** `robots.txt` y el middleware de §1 son complementarios, no redundantes. `robots.txt` disuade crawlers legítimos; el middleware de §1 corta el costo de los bots maliciosos que no respetan `robots.txt`.

### Archivos afectados

| Archivo | Cambio |
|---------|--------|
| `src/public/robots.txt` | Modificado — añadir `Allow: /` y `Disallow: /api/` |

## §3 — Subcollección `users/{userId}/auth`

### Contexto y deuda técnica

En v3, los tokens de Google OAuth2 se almacenan **embebidos** en el documento principal del usuario bajo la clave `auth`:

```
users/{userId}  →  { email, firstName, ..., auth: { googleToken, googleRefreshToken, refreshToken, apiToken } }
```

Se añadió `User.toPublic()` como parche para que ninguna respuesta API exponga esos tokens. Esta solución funciona pero es frágil: depende de que todo route handler recuerde llamar a `toPublic()`, y mantiene credenciales en el mismo documento que datos de perfil.

El patrón correcto ya existe en el proyecto: los API Keys viven en `users/{userId}/apiKeys` (subcollección), implementado en v4 §3. Esta sección aplica el mismo patrón a los tokens OAuth de Google.

### Campos afectados

| Campo en User.auth | Estado | Acción |
|--------------------|--------|--------|
| `googleToken` | Activo — escrito por OAuth callback | Mover a subcollección |
| `googleRefreshToken` | Activo — escrito por OAuth callback | Mover a subcollección |
| `refreshToken` | **Campo muerto** — nunca escrito en el código actual | Eliminar de modelo, parser, schema y Firestore |
| `apiToken` | **Eliminado en v4 §3.5** — campo muerto en Firestore | Limpiar de modelo, parser, schema si persiste |

### Diseño de la subcollección

- Ruta: `users/{userId}/auth`
- Un único documento con id `google` que contiene los tokens de la sesión OAuth de Google.
- Estructura del documento:

| Campo | Tipo | Descripción |
|-------|------|-------------|
| `googleToken` | string | Access token de Google OAuth2 (corta duración) |
| `googleRefreshToken` | string | Refresh token de Google OAuth2 (larga duración) |
| `updatedAt` | Timestamp | Última escritura (set manual en `AuthTokenService`) |

### Nuevo servicio: `AuthTokenService`

- Ruta: `src/api/users/services/auth-token.service.js`
- No extiende `CrudService` — igual que `ApiKeyService`, la ruta de la subcollección es dinámica.
- Métodos:
  - `read(userId)` — lee `users/{userId}/auth/google`; devuelve `null` si no existe
  - `write(userId, tokens)` — escribe (set con merge) `users/{userId}/auth/google`

### Cambios en cascada

| Archivo | Cambio |
|---------|--------|
| `src/api/users/services/auth-token.service.js` | Nuevo — `AuthTokenService` |
| `src/utils/auth/strategies/google-oauth2.strategy.js` | Usa `AuthTokenService.write()` en lugar de `userService.update()` para tokens |
| `src/api/users/models/user.model.js` | Eliminar params `googleToken`, `googleRefreshToken`, `refreshToken`, `apiToken` del constructor y `this.auth`; eliminar `toPublic()` |
| `src/api/users/parsers/user.parser.js` | Eliminar `auth` de `createUserParser` y `updateUserParser` |
| `src/api/users/schemas/user.schema.js` | Eliminar sub-schema `auth` |
| `src/api/users/routes/user.route.api.js` | Reemplazar `u.toPublic()` / `data.toPublic()` por objeto plano con los campos públicos |
| `src/api/auth/routes/auth.route.api.js` | Reemplazar `req.user.toPublic()` por objeto plano |
| `src/api/users/models/__test__/user.model.test.js` | Actualizar tests de `toPublic()` (método eliminado) y constructor |

### Migración de datos existentes

El documento `users/{userId}` en Firestore existente puede contener `auth: {}` con tokens ya expirados. Opciones:

- **Opción A (recomendada):** No limpiar activamente. Los parsers simplemente dejan de leer `auth` del documento principal. Los tokens expirados quedan como campos huérfanos sin impacto funcional; se limpiarán naturalmente si en el futuro se hace una migración batch o se accede a los usuarios individualmente.
- **Opción B:** Script de migración que aplica `FieldValue.delete()` sobre `auth` en todos los documentos de `users`. Añade complejidad operativa sin beneficio inmediato dado que los tokens de acceso de Google expiran en ~1 hora.

La elección entre A y B queda abierta para el backend-engineer. Documentar la decisión tomada en el commit `[feat]`.

## §4 — Integridad referencial en la capa de datos

### Contexto y deuda técnica

La capa de datos tiene tres gaps de consistencia causados por la naturaleza no-relacional de Firestore:

| Gap | Descripción | Consecuencia |
|-----|-------------|--------------|
| **G1** | Al eliminar un usuario, sus redirects quedan huérfanos | Solo admin puede verlos/eliminarlos |
| **G2** | Al hacer PATCH de `User.groups`, los documentos de Group no se actualizan | `Group.users` queda desincronizado con `User.groups` |
| **G3** | Al eliminar un grupo, las entradas `{scope}:{slug}` en `Redirect.permission` no se limpian | Si se crea un nuevo grupo con el mismo slug, heredan acceso a redirects del grupo anterior |

G3 también resuelve el problema de **reutilización de slugs**: la limpieza de permisos al eliminar un grupo garantiza que un slug liberado no transfiere acceso a futuros grupos con el mismo nombre.

### Patrón arquitectónico

El patrón **batch-first** ya existe en `GroupService.update()` y `UserService.delete()`: se crea un `WriteBatch`, se inscriben todas las operaciones necesarias (documento principal + cascades), y se hace un único `batch.commit()`. Esta sección aplica el mismo patrón a los gaps identificados, sin cambios en `CrudService`.

Los cascades cross-resource (entre usuarios/grupos y redirects) se centralizan en un nuevo `CascadeService`. Los cascades dentro del dominio usuario↔grupo continúan en `MembershipService` con un nuevo método.

### Comportamiento especificado por escenario

#### G1 — Usuario eliminado → redirects

Cuando se elimina un usuario, todos los redirects cuyo campo `owner` coincide con el email del usuario se modifican en un único `WriteBatch`:

- `owner`: se elimina el campo (`FieldValue.delete()`)
- `permission`: se vacía el array (`[]`)

El redirect queda en Firestore sin owner y sin permisos — visible únicamente para admin. Un futuro servicio de GC (fuera del alcance de v4.1) limpiará estos redirects huérfanos tras un periodo configurable.

La operación de eliminación de usuario ya construye un `WriteBatch` (para limpiar group membership). Las operaciones sobre redirects se añaden al mismo batch antes del commit.

#### G2 — PATCH de `User.groups` → sync de documentos Group

Cuando `UserService.update()` recibe un payload que incluye `groups`, la operación:

1. Hace fetch-first del documento actual del usuario para obtener `oldGroups`.
2. Construye un `WriteBatch` con la actualización del documento del usuario.
3. Calcula el diff (`addedSlugs`, `removedSlugs`) y añade al batch las operaciones `FieldValue.arrayUnion(userId)` / `FieldValue.arrayRemove(userId)` en cada documento de Group afectado.
4. Hace `batch.commit()`.

Esta lógica la provee el nuevo método `MembershipService.addOpsToSyncUserGroups(batch, userId, oldSlugs, newSlugs)`.

Si el payload de PATCH no incluye `groups`, `UserService.update()` delega en `super.update()` (comportamiento actual sin cambio).

#### G3 — Grupo eliminado → permisos en redirects

Cuando se elimina un grupo (slug `S`), todos los redirects que contengan cualquiera de `read:S`, `edit:S` o `delete:S` en su array `permission` deben actualizarse para eliminar esas entradas.

La consulta Firestore usa `array-contains-any` con los tres valores (`['read:S', 'edit:S', 'delete:S']`) para obtener todos los redirects afectados en una sola query. Cada redirect encontrado recibe un `batch.update` con `FieldValue.arrayRemove('read:S', 'edit:S', 'delete:S')`.

La operación de eliminación de grupo ya construye un `WriteBatch` (para limpiar `User.groups`). Las operaciones sobre redirects se añaden al mismo batch.

### Nuevo servicio: `CascadeService`

- Ruta: `src/utils/cascade.service.js`
- No extiende `CrudService`. Accede a Firestore directamente vía `firestoreClient`.
- Depende únicamente de la colección `redirects` — sin dependencia en `UserService` ni `GroupService`.
- Métodos:
  - `addOpsToOrphanUserRedirects(batch, ownerEmail)` — consulta `redirects` donde `owner == ownerEmail`; añade al batch un update por cada resultado que elimina `owner` y vacía `permission`. No hace commit.
  - `addOpsToCleanGroupPermissions(batch, slug)` — consulta `redirects` donde `permission array-contains-any ['read:{slug}', 'edit:{slug}', 'delete:{slug}']`; añade al batch un `FieldValue.arrayRemove` por cada resultado. No hace commit.

### Expansión de `MembershipService`

Nuevo método `addOpsToSyncUserGroups(batch, userId, oldSlugs, newSlugs)`:
- Calcula `addedSlugs = newSlugs - oldSlugs` y `removedSlugs = oldSlugs - newSlugs`.
- Para cada slug añadido: resuelve el Group via `GroupService.getBySlug(slug)`, añade `batch.update` con `FieldValue.arrayUnion(userId)`.
- Para cada slug eliminado: resuelve el Group via `GroupService.getBySlug(slug)`, añade `batch.update` con `FieldValue.arrayRemove(userId)`.
- No hace commit — el caller es responsable.

### Inyección de dependencias

`CascadeService` se inyecta en:
- `UserService` (constructor) — usado en `delete()` y en el override de `update()`
- `GroupService` (constructor) — usado en `delete()`

El punto de cableado es el mismo archivo de rutas que ya instancia las cadenas de servicios (`user.route.api.js`, `group.route.api.js`). Se crea una instancia única de `CascadeService` y se pasa a ambos.

### Limitación: WriteBatch ≤ 500 operaciones

Firestore `WriteBatch` admite un máximo de 500 operaciones por commit. Si un usuario tiene más de ~490 redirects (margen para las ops de group cleanup), la operación atómica excede el límite.

Para v4.1: documentar la limitación y aceptarla — el volumen actual del proyecto no la alcanza. Si en el futuro el volumen crece, la solución es ejecutar batches paginados en secuencia (no atómicos entre páginas).

| `src/api/users/services/membership.service.js` | Añadir `addOpsToSyncUserGroups` |
| `src/api/users/services/user.service.js` | `delete()`: integrar `CascadeService`; `update()`: override batch-first cuando `groups` cambia |
| `src/api/groups/services/group.service.js` | `delete()`: integrar `CascadeService` |
| `src/api/users/routes/user.route.api.js` | Cablear `CascadeService` en la instanciación |
| `src/api/groups/routes/group.route.api.js` | Cablear `CascadeService` en la instanciación |

## §5 — Servicios como singletons — contenedor de dependencias

### Contexto y deuda técnica

Cada archivo de rutas y middleware instancia sus propios servicios. El resultado es que la misma clase se instancia múltiples veces en el proceso:

| Servicio | Instancias actuales | Archivos |
|----------|---------------------|----------|
| `UserService` | 6 | `group.route.api.js`, `redirect.route.api.js`, `user.route.api.js` (×2), `authenticate.middleware.js`, `google-oauth2.strategy.js` |
| `GroupService` | 3 | `group.route.api.js`, `redirect.route.api.js`, `user.route.api.js` |
| `ApiKeyService` | 2 | `api-key.route.js`, `authenticate.middleware.js` |
| `MembershipService` | 1 | `user.route.api.js` |
| `RedirectServiceApi` | 1 | `redirect.route.api.js` |
| `RedirectService` | 1 | `redirect.router.js` |

El problema no es de conexiones (el cliente Firestore ya es singleton en `src/lib/firestore.js`), sino de cableado disperso: la cadena de inyección de dependencias (`userServiceForGroup → groupService → membershipService → userService`) se repite en distintos archivos con variaciones sutiles. Con el `CascadeService` de §4 esto se agrava — cada route file que necesite cascade tendría que replicar el cableado completo.

### Solución: contenedor de servicios

Crear `src/lib/services.js` — un módulo que instancia todos los servicios en el orden correcto y los exporta. Node.js cachea los módulos en `require()`, por lo que todos los consumers reciben la misma instancia sin ninguna infraestructura adicional.

El orden de instanciación respeta la cadena de dependencias existente:

```
userServiceForGroup  (UserService sin membershipService)
        ↓
groupService         (GroupService ← userServiceForGroup)
        ↓
membershipService    (MembershipService ← userServiceForGroup, groupService)
        ↓
userService          (UserService ← membershipService)

cascadeService       (CascadeService — sin deps de servicios de usuario/grupo)
redirectServiceApi   (RedirectServiceApi — independiente)
redirectService      (RedirectService — independiente)
apiKeyService        (ApiKeyService — independiente)
authTokenService     (AuthTokenService — independiente, ver §3)
```

`userServiceForGroup` se mantiene como instancia separada (no exportada como `userService`) para no introducir la dependencia circular que el patrón actual ya evita.

### Consumers que migran

Todos los archivos que hoy instancian servicios localmente pasan a importar desde `src/lib/services.js`:

| Archivo | Servicio(s) que importa |
|---------|------------------------|
| `src/api/users/routes/user.route.api.js` | `userService`, `groupService`, `membershipService` |
| `src/api/groups/routes/group.route.api.js` | `groupService`, `userService` |
| `src/api/redirect/routes/redirect.route.api.js` | `redirectServiceApi`, `groupService` |
| `src/api/users/routes/api-key.route.js` | `apiKeyService` |
| `src/redirect/routes/redirect.router.js` | `redirectService` |
| `src/middleware/authenticate.middleware.js` | `apiKeyService`, `userService` |
| `src/utils/auth/strategies/google-oauth2.strategy.js` | `userService`, `authTokenService` (tras §3) |

## §6 — Suite de tests E2E

### Contexto

Los tests unitarios actuales mockean Firestore. Antes de cada despliegue a producción se necesita verificar el stack completo: cliente HTTP → Express → middleware → servicio → Firestore real → respuesta. Esta suite cubre ese gap.

### Enfoque

- Tests corren contra `localhost:3000` con el servidor ya levantado (`npm run dev`).
- Firestore real — misma base de datos que desarrollo local (requiere `gcloud auth application-default login`).
- Runner: Jest con `--runInBand` — ejecución secuencial para evitar condiciones de carrera en Firestore.
- HTTP client: `fetch` nativo (Node 24).
- Los tests E2E **no forman parte de `npm test`** — son un paso de pre-deploy explícito.

```bash
npm run test:e2e   # requiere servidor corriendo en otro terminal
```

### Datos de prueba

- Todos los recursos de test se identifican con el dominio `@e2e.test` en emails y prefijo `e2e-` en slugs y paths.
- Cada suite limpia sus propios datos en `afterAll` — primero los recursos dependientes (redirects, API keys), luego usuarios y grupos.
- Si la suite se interrumpe antes del teardown, los datos huérfanos se pueden limpiar manualmente con un helper `npm run test:e2e:cleanup`.

### Autenticación en tests

El flujo OAuth2 de Google requiere interacción de browser — no es automatizable en E2E. En su lugar, los tests generan JWTs directamente usando `sign()` de `src/utils/auth/jwt.js` con el `JWT_SECRET` del `.env` local. Esto valida el comportamiento de autenticación/autorización sin depender de Google.

Los tests de Google OAuth2 (`GET /api/v1/auth/google` y callback) quedan fuera del alcance de esta suite.

### Estructura

```
e2e/
  helpers/
    auth.js       — genera JWTs para admin y usuarios regulares de test
    seed.js       — crea usuarios, grupos, redirects de test vía API o Firestore directo
    teardown.js   — elimina todos los recursos de test; exportado como script standalone
    client.js     — wrapper de fetch con base URL y headers comunes
  redirects.e2e.js
  users.e2e.js
  groups.e2e.js
  api-keys.e2e.js
```

### Cobertura por superficie

#### Redirects (`redirects.e2e.js`)

| Caso | Método | Verificación |
|------|--------|--------------|
| Admin crea redirect raíz | `POST /api/v1/redirects` | 201 + path correcto |
| Usuario regular crea redirect bajo su grupo | `POST /api/v1/redirects` | 201 |
| Usuario regular intenta crear redirect fuera de su grupo | `POST /api/v1/redirects` | 403 |
| Catch-all resuelve redirect creado | `GET /{path}` | 302 → URL destino |
| Catch-all retorna 404 para path inexistente | `GET /e2e-noexiste` | 404 |
| Listar redirects — admin ve todos | `GET /api/v1/redirects` | incluye redirects de otros usuarios |
| Listar redirects — usuario ve solo los suyos y los de su grupo | `GET /api/v1/redirects` | filtrado correcto |
