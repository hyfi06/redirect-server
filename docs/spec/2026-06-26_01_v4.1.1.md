# Spec v4.1.1 — Limpieza de código muerto y análisis DRY

**Fecha:** 2026-06-26
**Rama base:** dev (v4.1 completa)
**Estado al iniciar v4.1.1:** API v1 completa, autenticada, suite E2E passing. Todos los ítems de v4.1 (bot-reject, robots.txt, auth subcollección, soft-delete, singletons, E2E) cerrados y en producción (rama dev).

---

## Contexto y motivación

v4.1.1 es un sprint de mantenimiento puro. Sin features nuevas. Dos objetivos:

1. **Eliminar código muerto:** `AuthTokenService` y su subcollección `users/{userId}/auth` nunca se leen. Escribir tokens de Google en Firestore en cada login OAuth tiene costo de escritura sin ningún beneficio funcional.

2. **Reducir duplicación:** seis patrones repetidos identificados por análisis estático del código. Prioridad asignada a cada uno según impacto en mantenibilidad.

---

## §1 — Eliminar subcollección `users/{userId}/auth`

### Motivación y decisión

En v4.1 §3 se diseñó `AuthTokenService` para separar los tokens de Google OAuth de los campos de perfil del usuario, usando la subcollección `users/{userId}/auth/google`. El diseño fue correcto en ese contexto (reemplazaba tokens embebidos en el documento principal).

Sin embargo, **ningún path de código lee estos tokens**. El servidor usa Google OAuth exclusivamente como mecanismo de autenticación para emitir un JWT propio. Una vez emitido el JWT:

- El `accessToken` de Google (duración ~1 hora) ya no se usa para nada.
- El `refreshToken` de Google no se usa — el servidor no renueva tokens de Google.
- La sesión del usuario se gestiona con el JWT propio firmado con `JWT_SECRET`.

El resultado es que `authTokenService.write()` en `google-oauth2.strategy.js` ejecuta una escritura de Firestore en cada login OAuth que no aporta ningún valor. `AuthTokenService.read()` existe pero ningún caller lo invoca.

**Decisión:** eliminar `AuthTokenService`, su subcollección, y la llamada a `write()` en la strategy OAuth.

### Datos existentes en Firestore

Los documentos `users/{userId}/auth/google` ya escritos en desarrollo quedan **huérfanos** — ningún código los leerá. No se limpian activamente en este spec:

- El `accessToken` de Google expira en ~1 hora; el `refreshToken` puede ser revocado por Google o el usuario en cualquier momento.
- No hay valor en leerlos ni en borrarlos activamente.
- Si en el futuro se necesita una migración batch de Firestore, se hace entonces.

La subcollección `auth` y su documento `google` permanecen en Firestore como datos huérfanos sin impacto funcional. El teardown E2E (`e2e/helpers/teardown.js`) sí debe actualzarse para no intentar limpiar una subcollección que el código ya no gestiona.

## §2 — Análisis DRY del proyecto

El análisis cubre los seis patrones de duplicación identificados en el código. Cada ítem incluye los archivos exactos, la línea aproximada de inicio, el impacto real en mantenibilidad, y la propuesta de solución. Los ítems están ordenados de mayor a menor prioridad de implementación.

---

### §2.1 — Middleware de rechazo de API Keys duplicado

**Patrón:** Ambos routers de gestión (usuarios y grupos) tienen el mismo bloque de middleware anónimo que rechaza requests autenticados con API Key.

**Dónde aparece:**

| Archivo | Líneas aproximadas |
|---------|-------------------|
| `src/api/users/routes/user.route.api.js` | 40–43 |
| `src/api/groups/routes/group.route.api.js` | 21–24 |

El bloque es idéntico en ambos: comprueba `req.user.apiKey !== undefined` y llama `next(boom.forbidden('API Keys cannot be used on this resource'))`. La lógica, el mensaje de error, y el formato del cheque son idénticos carácter a carácter.

**Impacto en mantenibilidad:** Medio. Si el mensaje de error o la condición cambia (por ejemplo, si en el futuro se permiten API Keys con scope `admin:users`), el cambio debe aplicarse en dos lugares. Existe riesgo de divergencia silenciosa.

**Propuesta de solución:** Extraer a un middleware nombrado `requireJwt` en un nuevo archivo `src/middleware/require-jwt.middleware.js`. Ambos routers importan el middleware y lo montan con `routerApi.use(requireJwt)`. La semántica del nombre ("esta ruta requiere JWT, no API Key") es más expresiva que un bloque anónimo inline.

**Prioridad:** Alta — es el cambio más pequeño con el beneficio más claro.

**Checkboxes:**
- [x] §2.1.1: crear `src/middleware/require-jwt.middleware.js` con la lógica de rechazo de API Keys
- [x] §2.1.2: crear `src/middleware/__test__/require-jwt.middleware.test.js` con tests de las dos ramas (API Key → 403, JWT → pass)
- [x] §2.1.3: reemplazar los bloques inline en `user.route.api.js` y `group.route.api.js` por `routerApi.use(requireJwt)`

### §2.2 — `findInactive()` duplicado en UserService y GroupService

**Patrón:** El método `findInactive(options)` en `UserService` y `GroupService` es carácter-a-carácter idéntico. Ambas implementaciones consultan `where('deletedAt', '!=', null)`, ordenan por `deletedAt desc`, y aplican `offset`/`limit`.

**Dónde aparece:**

| Archivo | Líneas aproximadas |
|---------|-------------------|
| `src/api/users/services/user.service.js` | 91–101 |
| `src/api/groups/services/group.service.js` | 78–88 |

**Impacto en mantenibilidad:** Medio. Cualquier cambio en la semántica de soft-delete (por ejemplo, añadir un campo de razón de borrado, cambiar el ordenamiento, o añadir paginación por cursor) debe aplicarse en dos lugares. El riesgo actual es bajo, pero crece con cada recurso soft-deletable que se añada.

**Propuesta de solución:** Mover `findInactive(options)` a `CrudService` en `src/utils/crud.service.js`. Tanto `UserService` como `GroupService` extienden `CrudService` y ya tienen acceso a `this.db.collection` y `this.docParser`. Ambos heredarían el método sin modificación. Los recursos que no soporten soft-delete nunca llaman al método, por lo que no hay riesgo de comportamiento incorrecto.

**Prioridad:** Alta — el patrón de herencia ya existe; la extracción es directa.

**Checkboxes:**
- [x] §2.2.1: mover `findInactive(options)` a `src/utils/crud.service.js` como método de la clase base
- [x] §2.2.2: eliminar los métodos duplicados de `src/api/users/services/user.service.js` y `src/api/groups/services/group.service.js`
- [x] §2.2.3: actualizar `src/utils/__test__/crud.service.test.js` para cubrir `findInactive` en la clase base (paginación, resultado vacío, docs con `deletedAt != null`)
- [x] §2.2.4: verificar que los tests existentes de `user.service.test.js` y `group.service.test.js` relacionados con `findInactive` siguen pasando (sin modificación si ya mockean CrudService correctamente)

### §2.3 — Campo público del usuario definido en dos lugares

**Patrón:** El conjunto de campos públicos del modelo `User` está definido dos veces en el código:

- `src/api/users/routes/user.route.api.js` líneas 20–32: función `toPublic(user)` que devuelve un objeto con 9 campos (`id`, `email`, `firstName`, `lastName`, `groups`, `role`, `deletedAt`, `created`, `updated`).
- `src/api/auth/routes/auth.route.api.js` línea 23: destructuring manual del mismo objeto, pero con **8 campos** — omite `deletedAt`.

La omisión de `deletedAt` en la respuesta OAuth (`GET /api/v1/auth/google/callback`) es una inconsistencia real: un usuario recién autenticado via OAuth recibe un objeto de usuario en el JWT response sin `deletedAt`, mientras que `GET /api/v1/users/me` sí lo incluye.

**Impacto en mantenibilidad:** Medio. Si se añade un campo al modelo público del usuario (por ejemplo, `avatarUrl`, `lastLoginAt`), debe añadirse en dos archivos de rutas distintos. La inconsistencia actual de `deletedAt` ya es un bug potencial en la respuesta de login.

**Propuesta de solución:** Extraer a un helper compartido `src/api/users/utils/user-public.js` que exporte `toPublic(user)`. Ambos routers importan la función desde esa ubicación. Los routers no se importan entre sí — la util es el único punto de acoplamiento. La función en `user.route.api.js` se elimina y se reemplaza por el import.

**Prioridad:** Media — la inconsistencia de `deletedAt` es el driver principal, más allá del DRY.

**Checkboxes:**
- [x] §2.3.1: crear `src/api/users/utils/user-public.js` con la función `toPublic(user)` que incluya los 9 campos (incluyendo `deletedAt`)
- [x] §2.3.2: actualizar `src/api/users/routes/user.route.api.js` para importar `toPublic` desde el nuevo archivo y eliminar la definición local
- [x] §2.3.3: actualizar `src/api/auth/routes/auth.route.api.js` para importar `toPublic` y reemplazar el destructuring manual (esto corrige la inconsistencia de `deletedAt`)
- [x] §2.3.4: actualizar tests de `user.route.api.test.js` y `auth.route.api.test.js` para verificar que `deletedAt` aparece en ambas respuestas

### §2.4 — Patrón de control de acceso a redirects repetido en tres handlers

**Patrón:** Los handlers `GET /:id`, `PATCH /:id`, y `DELETE /:id` del router de redirects contienen variaciones del mismo patrón de acceso:

1. Construir array de permisos: `req.user.groups.map(g => '{scope}:{g}')` con scope distinto (`read`, `edit`, `delete`).
2. Verificar: `req.user.role === 'admin' || resource.owner === req.user.userId || resource.permission.some(p => scopes.includes(p))`.
3. Si falla: `next(boom.forbidden('Insufficient permissions'))`.

**Dónde aparece:**

| Archivo | Handler | Líneas aproximadas | Scope |
|---------|---------|-------------------|-------|
| `src/api/redirect/routes/redirect.route.api.js` | `GET /:id` | 71–76 | `read` |
| `src/api/redirect/routes/redirect.route.api.js` | `PATCH /:id` | 136–141 | `edit` |
| `src/api/redirect/routes/redirect.route.api.js` | `DELETE /:id` | 162–168 | `delete` |

**Impacto en mantenibilidad:** Medio. Tres repeticiones en el mismo archivo — la proximidad reduce el riesgo de divergencia, pero cualquier cambio en la lógica de acceso (por ejemplo, añadir un rol `moderator`) debe aplicarse en tres handlers.

**Propuesta de solución:** Extraer a una función helper local en el mismo archivo (no requiere un módulo separado):

```
canAccess(user, resource, scope) → boolean
```

Recibe el objeto `user` de `req.user`, el documento `resource` de Firestore, y el scope (`'read'`, `'edit'`, `'delete'`). Los tres handlers usan `if (!canAccess(req.user, existing, 'edit')) return next(boom.forbidden(...))`. La función permanece en el archivo de rutas — no cruza módulos.

**Prioridad:** Media — el impacto es acotado al mismo archivo; la extracción mejora la legibilidad.

**Checkboxes:**
- [x] §2.4.1: extraer función `canAccess(user, resource, scope)` como helper local en `src/api/redirect/routes/redirect.route.api.js`
- [x] §2.4.2: reemplazar los tres bloques de control de acceso por llamadas a `canAccess`
- [x] §2.4.3: actualizar `src/api/redirect/routes/__test__/redirect.route.api.test.js` para cubrir el helper de forma indirecta a través de los handlers (no requiere tests unitarios separados del helper)

### §2.5 — Conversión de Timestamps duplicada en los tres docParsers

**Patrón:** Los tres `docParser` convierten Firestore Timestamps a `Date` usando la misma expresión:

- `created: new Date(data.created.toMillis())`
- `updated: new Date(data.updated.toMillis())`
- `deletedAt: data.deletedAt ? new Date(data.deletedAt.toMillis()) : null` (user y group)

**Dónde aparece:**

| Archivo | Líneas aproximadas |
|---------|-------------------|
| `src/api/redirect/parsers/redirect.parser.js` | 20–21 |
| `src/api/users/parsers/user.parser.js` | 18–21 |
| `src/api/groups/parsers/group.parser.js` | 13–17 |

**Impacto en mantenibilidad:** Bajo. La conversión es trivial y estable. No hay riesgo real de divergencia. El beneficio es principalmente de legibilidad y de documentar el contrato "Timestamp → Date" de forma explícita.

**Propuesta de solución:** Añadir dos funciones de utilidad a `src/utils/clean.data.utils.js`:
- `parseTimestamp(ts)` — convierte un Firestore Timestamp a `Date`. Complementa a `deleteRegData` y `cleanDocObject`, que ya viven en ese archivo.
- `parseOptionalTimestamp(ts)` — devuelve `null` si `ts` es falsy, `Date` si no. Usada para `deletedAt`.

Los tres parsers importan las funciones desde `clean.data.utils.js`.

**Prioridad:** Baja — el cambio es cosmético. Implementar solo si el sprint tiene capacidad sobrante.

**Checkboxes:**
- [ ] §2.5.1: añadir `parseTimestamp(ts)` y `parseOptionalTimestamp(ts)` a `src/utils/clean.data.utils.js`
- [ ] §2.5.2: actualizar los tres docParsers para usar las funciones de utilidad
- [ ] §2.5.3: añadir tests de las dos funciones en `src/utils/__test__/clean.data.utils.test.js`

### §2.6 — Definiciones Joi comunes repetidas en los tres schemas

**Patrón:** Los campos `offset`, `limit`, `orderBy`, e `inactive` se definen con idénticas restricciones en los tres archivos de schema. El campo `id` se define como `Joi.string()` en redirect y user schemas.

**Dónde aparece:**

| Campo | redirect.schema.js | user.schema.js | group.schema.js |
|-------|--------------------|----------------|-----------------|
| `id` | línea 3 | línea 3 | — (inline en `idParamSchema`) |
| `offset` | línea 19 | `getUsersQuerySchema` inline | `getGroupQuerySchema` inline |
| `limit` | línea 20 | `getUsersQuerySchema` inline | `getGroupQuerySchema` inline |
| `orderBy` | línea 18 | — | `getGroupQuerySchema` inline |
| `inactive` | — | línea 13 | línea 22 |

**Impacto en mantenibilidad:** Bajo. Las definiciones Joi son pequeñas y rara vez evolucionan. No hay riesgo de comportamiento incorrecto.

**Propuesta de solución:** Crear `src/api/schemas/common.schema.js` que exporte los campos compartidos. Cada schema de recurso importa lo que necesita. Reduce la superficie de búsqueda cuando alguien quiere cambiar, por ejemplo, el `min(1)` de `offset`.

**Prioridad:** Baja — implementar solo si el sprint tiene capacidad sobrante.

**Checkboxes:**
- [ ] §2.6.1: crear `src/api/schemas/common.schema.js` con `id`, `offset`, `limit`, `orderBy`, `inactive`
