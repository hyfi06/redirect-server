# Plan — Hotfix: `errorHandler` debe devolver JSON para rutas API

**Fecha:** 2026-06-23
**Estado:** CERRADO
**Rama:** `hotfix-api-404-json`
**Diagnóstico de referencia:** `docs/diag/2026-06-23_01_error-handler-api-404-html.md`

---

## Contexto

El `errorHandler` de `src/middleware/error.handler.js` sirve HTML (`NotFound.html`, `serverError.html`) para todos los errores 404 y 500-producción, sin distinguir entre rutas de browser y rutas de API. Para las rutas `/api/v1/**`, el contrato de `docs/api/v1.md` exige que todos los errores devuelvan JSON en formato Boom. Este plan corrige esa divergencia.

---

## §1 ✅ — Fix `errorHandler` para distinguir rutas API vs browser

**Archivo:** `src/middleware/error.handler.js`

El backend-engineer debe modificar `errorHandler` para que, antes de decidir el formato de respuesta, determine si la petición proviene de una ruta API. El criterio de clasificación debe ser: si `req.path` comienza con `/api/`, la respuesta debe ser JSON con el payload Boom, independientemente del status code. Solo si la petición NO es de API se aplica la lógica HTML actual para 404 y 500-producción.

Restricciones de diseño:

- El criterio de detección de ruta API debe ser robusto frente a futuros prefijos (p.ej. `/api/v2/`). Usar `/api/` como prefijo es suficiente para el estado actual; el backend-engineer puede usar `req.path.startsWith('/api/')` o `req.originalUrl.startsWith('/api/')` según lo que Express exponga con fiabilidad en ese punto del pipeline.
- La lógica HTML existente (404 → `NotFound.html`, 500-prod → `serverError.html`) debe preservarse intacta para peticiones no-API.
- El campo `stack` en la respuesta JSON API debe seguir la misma condición que la rama `else` actual: solo en `config.dev`.
- No introducir dependencias nuevas.

**Commit esperado:**

```

plan: 2026-06-23_01_hotfix-api-404-json §1
- API/browser detection and JSON branch in src/middleware/error.handler.js

```

## §2 ✅ — Actualizar tests de `error.handler`

**Archivo:** `src/middleware/__test__/error.handler.test.js`

Los tests actuales usan un `mockReq = {}` sin `path` ni `originalUrl`, por lo que no ejercitan el nuevo comportamiento API/browser. El test-engineer debe:

- Añadir casos que simulen peticiones a rutas API (`req.path = '/api/v1/...'`) y verificar que 404 devuelve JSON, no `sendFile`.
- Añadir casos que simulen peticiones a rutas browser (`req.path = '/some-path'`) y verificar que 404 devuelve `sendFile`, no JSON.
- Cubrir la rama 500-producción bajo ruta API (debe devolver JSON) y bajo ruta browser (debe devolver `sendFile`).
- Mantener los casos existentes. Si algún caso existente falla porque `mockReq = {}` ahora produce un comportamiento diferente al esperado, el test-engineer debe adaptarlo para que `req.path` refleje explícitamente el tipo de ruta que el test pretende cubrir.

No eliminar tests existentes sin justificación explícita en el commit.
