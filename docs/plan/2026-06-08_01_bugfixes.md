# Plan — Bugfixes: wrapErrors double-call y RedirectServiceApi non-404 swallow

**Rama:** `v3-beta`  
**Origen:** Encontrados durante la sesión de cobertura de tests post-Bloque 4.

---

## Contexto

Durante la revisión de cobertura de tests se detectaron dos bugs en código de producción existente. Ninguno está bloqueado por trabajo en vuelo — son correcciones independientes. Cada bug sigue el ciclo completo `[fix] → [test] → [docs]`.

---

## Bug 1 — `wrapErrors()` llama `next()` dos veces para errores no-Boom

**Archivo:** `src/middleware/error.handler.js`

**Código actual (líneas 18-23):**
```js
function wrapErrors(err, req, res, next) {
  if (!err.isBoom) {
    next(boom.badImplementation(err));  // ← falta return
  }
  next(err);                            // ← siempre se ejecuta
}
```

**Problema:** Para errores no-Boom, `next()` se invoca dos veces: primero con el error wrapeado como boom 500, luego con el error original. Express ignora la segunda llamada si la respuesta ya fue enviada, pero en tests el mock de `next` registra dos invocaciones cuando solo debería tener una. El test existente `'should wrap non-boom errors'` no verifica `toHaveBeenCalledTimes`, por lo que el bug pasa desapercibido en la suite actual.

**Fix:**
```js
function wrapErrors(err, req, res, next) {
  if (!err.isBoom) {
    return next(boom.badImplementation(err));
  }
  next(err);
}
```

**Impacto en tests:** El test `'should wrap non-boom errors'` en `src/middleware/__test__/error.handler.test.js` debe agregar `expect(mockNext).toHaveBeenCalledTimes(1)` para verificar que el fix previene la segunda llamada.

## Bug 2 — `RedirectServiceApi.create()` swallows errores no-404

**Archivo:** `src/api/redirect/services/redirect.service.api.js`

**Código actual (líneas 40-48):**
```js
async create(redirect) {
  try {
    await this.getByPath(redirect.path.replace(/\/$/, ''));
  } catch (error) {
    const newDoc = await this.db.create(this.createParser(redirect));
    return this.docParser(newDoc);
  }
  throw boom.badRequest('Path already taken');
}
```

**Problema:** El `catch` captura todos los errores que lanza `getByPath`, no solo los 404. Si Firestore falla durante la verificación de unicidad (error de red, cuota, etc.), el servicio traga el error y procede a crear el documento de todas formas — pudiendo crear duplicados o registros corruptos. Este comportamiento está documentado como bug conocido en el test existente con el comentario `NOTE`.

**Comportamiento correcto — patrón de `GroupService.create()`:**
```js
async create(redirect) {
  try {
    await this.getByPath(redirect.path.replace(/\/$/, ''));
  } catch (e) {
    if (e.output?.statusCode !== 404) throw e;  // ← guarda que falta
    const newDoc = await this.db.create(this.createParser(redirect));
    return this.docParser(newDoc);
  }
  throw boom.badRequest('Path already taken');
}
```

**Impacto en tests:** El test `'proceeds to call db.create even when getByPath throws a non-404 Firestore error (current behavior)'` en `src/api/redirect/services/__test__/redirect.service.api.test.js` documenta el comportamiento buggy. Debe actualizarse para verificar que el error es relanzado y que `db.create` no se llama.

## Riesgos / Notas

**N1 — Orden de corrección independiente.** Los dos bugs no tienen dependencias entre sí. Se recomienda corregir Bug 1 primero (cambio de una línea, riesgo mínimo) y luego Bug 2.

**N2 — Bug 2: comportamiento de `GroupService.create()` es el patrón canónico.** La guarda `if (e.output?.statusCode !== 404) throw e` ya existe en `src/api/groups/services/group.service.api.js`. El fix es alinear `RedirectServiceApi` con ese patrón, no inventar uno nuevo.

**N3 — Bug 2: el test NOTE documenta el comportamiento buggy activo.** El test `'proceeds to call db.create even when getByPath throws a non-404 Firestore error (current behavior)'` pasará en rojo al aplicar el fix — es el indicador esperado de que el fix funciona. El test-engineer debe reemplazarlo, no simplemente borrarlo.

**N4 — Bug 1: impacto en producción bajo pero real.** Express sí ignora la segunda llamada a `next()` tras enviar la respuesta, pero el comportamiento es técnicamente incorrecto y viola el contrato de Express de "un único `next()` por middleware". Puede causar `Cannot set headers after they are sent` en entornos con logging o middlewares que sí procesan después de `errorHandler`.
