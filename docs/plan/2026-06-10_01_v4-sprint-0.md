## Objetivo

Sprint 0 limpia la base de código antes de que comience cualquier trabajo funcional de v4. Tiene dos propósitos:

1. **Eliminar la deuda estructural acumulada** (re-exports sin valor, nombres inconsistentes) para que el resto de sprints trabaje sobre una base coherente y no acumule más inconsistencias al crear archivos nuevos.
2. **Garantizar que los índices Firestore están versionados** para que una pérdida del proyecto GCP no deje la API rota silenciosamente en producción.

Al completarse Sprint 0, el repositorio debe tener los tests en verde, todos los imports apuntando a los nombres nuevos, y `scripts/manage-indexes.js` presente con `npm run indexes` operativo para sincronizar índices Firestore con `gcloud`.

## Pasos

### §4.1 — Eliminar re-exports de `src/redirect/`

Hay tres archivos en `src/redirect/` cuyo único contenido es `module.exports = require(...)` apuntando a su contraparte en `src/api/redirect/`. Son indirección sin abstracción: no añaden tipado, no adaptan interfaces, no tienen lógica. Eliminarlos reduce el número de nodos del grafo de dependencias y hace evidente dónde vive la lógica real.

El único consumidor de producción de estos re-exports es `src/redirect/routes/redirect.router.js`, que importa el servicio vía `'../services/redirect.service'`. El test de esa ruta (`src/redirect/routes/__test__/redirect.route.test.js`) lo mockea con la misma ruta relativa.

- [x] **Paso 1 — Actualizar el import en `redirect.router.js`.**
  En `src/redirect/routes/redirect.router.js`, cambiar el `require` del servicio para que apunte directamente a `src/api/redirect/services/redirect.service.api.js` (ruta relativa desde el router). Esto satisface la restricción de §4.1: el router importa la implementación real sin pasar por el re-export.

- [x] **Paso 2 — Actualizar el mock en el test del router.**
  En `src/redirect/routes/__test__/redirect.route.test.js`, el `jest.mock` apunta a `'../../services/redirect.service'` (el re-export). Actualizarlo para que apunte a la misma ruta que ahora usa el router de producción. Si el path del mock no coincide con el path del `require` en producción, Jest no interceptará el módulo y los tests fallarán. Este paso es bloqueante: si no se hace junto con el Paso 1, el test suite queda roto.

- [x] **Paso 3 — Eliminar los tres archivos de re-export.**
  Borrar:
  - `src/redirect/models/redirect.model.js`
  - `src/redirect/parsers/redirect.parsers.js`
  - `src/redirect/services/redirect.service.js`

  Verificar antes de borrar que ningún otro archivo del repositorio importa estas rutas (el grep de impacto no encontró otros consumidores, pero es buena práctica confirmarlo con un `grep -rn "redirect/models/redirect.model\|redirect/parsers/redirect.parsers\|redirect/services/redirect.service"` antes de eliminar).

El sufijo estándar del proyecto para estos artefactos es `.model.js`, `.parser.js` y `.service.js` sin infijos adicionales.

Esta tarea es un refactor puro: sin cambios de comportamiento, sin cambios de lógica, solo renombrado de archivos y actualización de todos sus importadores. El riesgo principal es dejar un import sin actualizar; el test suite lo detectará como `Cannot find module`.

### §1.4 — Índices Firestore versionados y script de gestión

El proyecto usa `gcloud` exclusivamente — no Firebase CLI. `firebase.json` no se crea. La sincronización de índices se gestiona con `scripts/manage-indexes.js`.

`firestore.indexes.json` ya existe en la raíz con los índices correctos. El gap pendiente es el script que permite sincronizarlos de forma reproducible.

- [x] **Paso 7 — Crear el directorio `scripts/` en la raíz del repositorio y el archivo `scripts/manage-indexes.js`.**

  El script debe cumplir exactamente el comportamiento especificado en §1.4 del spec:

  - Leer `firestore.indexes.json` de la raíz como fuente de verdad.
  - Obtener los índices actuales del proyecto con `gcloud firestore indexes composite list --format=json`.
  - Comparar por `collectionGroup + queryScope + fields`, ignorando `__name__`, `density`, `state`, y el nombre opaco.
  - Crear los índices del JSON que no existen en el proyecto con `gcloud firestore indexes composite create`.
  - Eliminar los índices obsoletos (existentes en el proyecto pero no en el JSON). Esta operación requiere confirmación interactiva; el flag `--force` la omite para CI/CD.
  - Tratar índices en estado `CREATING` como ya existentes — no duplicarlos.
  - Leer el proyecto GCP activo con `gcloud config get-value project` — no hardcodearlo.
  - Ser idempotente: sin cambios pendientes, dos ejecuciones consecutivas no producen ninguna operación.

  El script es un script Node.js puro que invoca `gcloud` mediante `child_process` (o equivalente). No introduce dependencias nuevas al proyecto.

- [x] **Paso 8 — Añadir `npm run indexes` a `package.json`.**

  En la sección `scripts` de `package.json`, añadir:

  ```
  "indexes": "node scripts/manage-indexes.js"
  ```

  El script `deploy` permanece sin cambios (`gcloud app deploy` detecta `app.yaml` automáticamente). `npm run indexes` y `npm run deploy` son comandos independientes — el desarrollador los corre en orden antes de un deploy que incluya cambios de índices.

- [x] **Paso 9 — Verificar que el `firestore.indexes.json` existente cubre los índices del spec.**

  El archivo actual tiene tres entradas: `redirects/owner+updated`, `redirects/permission+updated`, y `groups/slug+updated`. Las dos primeras son exactamente las requeridas por §1.4. La tercera (groups) es un índice existente no documentado en el spec — no eliminarla; es una mejora, no una regresión.

  El índice para `apiKeys` COLLECTION_GROUP (§3.7) pertenece a Sprint 4 y no debe añadirse aquí. Sprint 0 solo versiona lo que ya existe.

## Notas para el backend-engineer

**Orden de operaciones crítico:**

1. Los Pasos 1 y 2 (§4.1) deben ejecutarse juntos: si se actualiza el import en `redirect.router.js` sin actualizar el mock en el test, Jest dejará de interceptar el módulo y los tests de la ruta de redirect pasarán en silencio con el módulo real (sin mock), o fallarán con un error de instanciación de Firestore en el entorno de test. El Paso 3 (borrado de archivos) es seguro solo después de que los Pasos 1 y 2 estén en verde.

2. §4.1 antes de §4.2: el Paso 1 apunta el router a `redirect.service.api.js`. El Paso 4 renombra ese archivo a `redirect.service.js`. Esto significa que el Paso 4 incluye actualizar el import del router una segunda vez. No es un error — es la secuencia correcta: primero eliminar la indirección (§4.1), luego renombrar el destino (§4.2).

**Commits recomendados:**

El spec prescribe un commit por unidad de trabajo. Sprint 0 tiene tres unidades independientes — §4.1, §4.2 e §1.4 — que deben ir en commits separados de tipo `[refactor]` para §4.1 y §4.2, y `[chore]` para §1.4. No mezclar renombrados con la creación de `scripts/manage-indexes.js`.

**Test files — decisión pendiente:**

El spec §4.2 lista 10 archivos fuente a renombrar pero no menciona los archivos de test correspondientes (por ejemplo, `redirect.models.api.test.js` sigue con el nombre del archivo que ya no existe). El backend-engineer debe decidir si renombra los test files para mantener consistencia con sus nuevos nombres de sujeto, o los deja con el nombre anterior. Si los renombra, deben incluirse en el mismo commit `[refactor]` del §4.2. Si no los renombra, debe abrirse un ticket de seguimiento para no dejar deuda implícita. Recomendación: renombrarlos en el mismo commit para que el grafo archivo-test sea coherente.

**`firestore.indexes.json` no debe modificarse en Sprint 0:**

El archivo ya existe y es correcto. El único cambio en §1.4 es crear `scripts/manage-indexes.js` y añadir `npm run indexes` en `package.json`. No reescribir ni reformatear `firestore.indexes.json` — eso crearía un diff innecesario que ensucia el historial git de un archivo de infraestructura crítico.
