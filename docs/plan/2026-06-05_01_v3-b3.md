# Plan: Bloque 3 — CRUD de grupos (spec 2026-06-05_01_v3 §3)

## Observaciones previas a la implementación

- La colección Firestore `groups` ya está registrada en `src/config/index.js` como `config.firestore.collections.groups`.
- El patrón completo del recurso `users` se usa como referencia: model → schema → parsers → service → route → registro en index.
- `User.groups` almacena slugs (no IDs) — D1 del spec. El constructor de `User` ya acepta `groups: []`.
- La sincronización `Group.users` ↔ `User.groups` es bidireccional: cuando el admin modifica el array `users` de un grupo (PATCH), hay que calcular el diff (quién entró, quién salió) y actualizar `User.groups` en cada usuario afectado. Esto se hace en `GroupService.update()` de forma secuencial (no atómica — la atomicidad vía batch se difiere a v4 según el spec, D del bloque 3).
- El `slug` es inmutable (D7). El `updateParser` debe eliminarlo antes de escribir a Firestore. Además, si el body del PATCH incluye `slug`, la ruta debe responder 400 con `"slug is immutable"` — esto se hace en el handler antes de llegar al service, no en Joi (para dar un mensaje explícito).
- GET `/api/v1/groups` tiene semántica diferente según rol:
  - Admin: devuelve todos los grupos.
  - Usuario regular: devuelve solo los grupos a los que pertenece (`slug in req.user.groups`).
- GET `/api/v1/groups/:id` tiene la misma restricción: un usuario regular solo puede ver un grupo si pertenece a él.
- `CrudService.find()` recibe el query como array (`where(...query)`) — para la query de admin (todos los grupos) se puede usar `getAll()`. Para usuarios regulares con grupos, se necesita filtrar por slug. La estrategia es:
  - Admin: usar `service.getAll({ orderBy, offset, limit })`.
  - Usuario con grupos: usar un query `where('slug', 'in', req.user.groups)` — Firestore soporta `in` con hasta 30 valores.
  - Usuario sin grupos: devolver array vacío directamente (sin query a Firestore).
- `CrudService.find()` toma el query como un array que se pasa a `where(...query)` — esto solo soporta un único `where(field, op, value)`. Para la query `in`, el array sería `['slug', 'in', slugsArray]`. Funciona correctamente.
- `UserServices` ya existe como instancia inyectable — `GroupService.update()` necesita actualizar usuarios, por lo que `GroupService` debe tener acceso a `UserServices`. Se inyecta via constructor o import directo. Se usará import directo (mismo patrón que en otros servicios que no tienen dependencias cruzadas, pero dado que `GroupService` necesita actualizar usuarios, se instancia `UserServices` dentro de `GroupService`). Esto es un patrón no establecido en el codebase — se documenta como decisión.
- El `docParser` de grupo convierte los `Timestamps` de Firestore a `Date`, igual que en users.
- `createGroupParser` no incluye `id`, `created`, `updated` (los añade `FireStoreAdapter.create()` automáticamente).
- `updateGroupParser` usa `cleanDocObject` y `deleteRegData` igual que `updateUserParser`, más `delete data.slug`.

## Decisiones de diseño identificadas (resueltas)

| # | Decisión | Resolución |
|---|---|---|
| B3-D1 | ¿Cómo se inyecta `UserServices` en `GroupService`? | **Constructor injection.** El router instancia `new GroupService(userService)`. Facilita mocking en tests sin necesidad de `jest.mock()` del módulo entero, hace explícito el acoplamiento de dominio, y facilita el refactor a `MembershipService` en v4 cambiando solo el sitio de composición. |
| B3-D2 | GET `/groups` para usuario sin grupos: ¿query vacío o array vacío directo? | **Devolver `[]` directamente.** Guard: `Array.isArray(req.user.groups) && req.user.groups.length > 0`. La query `where('slug', 'in', [])` lanza error del SDK antes de hacer ninguna llamada de red. |
| B3-D3 | PATCH con `slug` en body: ¿validar en Joi (excluirlo silenciosamente) o responder 400 explícito en handler? | **Verificación en handler antes de Joi.** Orden en el handler PATCH: 1) `validatorHandler(idSchema, 'params')`, 2) `authorize('admin')` — un no-admin con `slug` en body recibe 403, no 400, 3) handler async: `if (req.body.slug !== undefined) → next(boom.badRequest('slug is immutable'))`, 4) Joi inline del body. |

## Detalle de implementación por archivo

### `group.model.api.js`
```js
class Group {
  constructor({ id, name, slug, users, created, updated }) {
    this.id = id || null;
    this.name = name;
    this.slug = slug;
    this.users = users !== undefined ? users : undefined; // R6: undefined → cleanDocObject omite el campo en PATCH; [] vacía el grupo
    if (created) this.created = created;
    if (updated) this.updated = updated;
  }
}
```

### `group.schema.js`
```js
const createGroupSchema = Joi.object({
  name: Joi.string().required(),
  slug: Joi.string().lowercase().pattern(/^[a-z0-9-]+$/).required(),
  users: Joi.array().items(Joi.string().email()),
});

const updateGroupSchema = Joi.object({
  name: Joi.string(),
  users: Joi.array().items(Joi.string().email()),
});

const idSchema = Joi.object({ id: Joi.string().required() });
const getGroupQuerySchema = Joi.object({ orderBy, offset, limit });
```

### `group.parser.api.js`
- `groupDocParser`: Convierte `DocumentSnapshot → Group`, timestamps a `Date`.
- `createGroupParser`: Devuelve `{ name, slug, users }` (sin `id`; defaults manejados por el constructor).
- `updateGroupParser`: Usa `deleteRegData` + `delete data.slug` + `cleanDocObject`.

### `group.service.api.js`
- `getBySlug(slug)`: query `where('slug', '==', slug)`, lanza `boom.notFound` si vacío.
- `create(group)`: verifica unicidad via `getBySlug`, lanza `boom.badRequest('Slug already taken')` si existe.
- `update(group)`: 
  1. Obtiene el grupo actual con `findOne(group.id)` para tener el `users` previo.
  2. Calcula diff: `added = newUsers - oldUsers`, `removed = oldUsers - newUsers`.
  3. Para cada email en `added`: obtiene el `User`, añade el slug del grupo a `user.groups`, actualiza.
  4. Para cada email en `removed`: obtiene el `User`, quita el slug de `user.groups`, actualiza.
  5. Llama a `super.update(group)` (o directo a `this.db.update`) para persistir el grupo.
  - Si `users` no cambia en el body, no hay diff que calcular.

### `group.route.api.js`
- `router.use(authenticate)` — todos requieren JWT.
- `GET /`: admin → `service.getAll(options)`; usuario con grupos → `service.find(['slug', 'in', req.user.groups], options)`; sin grupos → `res.json([])`.
- `GET /:id`: `service.findOne(id)`; si usuario regular, verificar que `req.user.groups.includes(group.slug)`.
- `POST /`: `authorize('admin')` + `validatorHandler(createGroupSchema, 'body')` → `service.create(new Group(req.body))`.
- `PATCH /:id`: `authorize('admin')` + verificar `slug` en body (→ 400 explícito) + `validatorHandler(updateGroupSchema, 'body')` → `service.update(new Group({ id, ...req.body }))`.
- `DELETE /:id`: `authorize('admin')` + `service.delete(id)`.

**Nota sobre GET `/` y `getAll`:** `CrudService.getAll()` no acepta `Filter` — sirve solo para admin (todos los docs). Para usuarios, se usa `find()` con `where('slug', 'in', slugsArray)`. El `find()` actual en `CrudService` llama a `fsQuery.where(...query)` — si el tercer elemento del array es un array de slugs, Firestore lo interpretará como `in` query.

**Nota sobre PATCH y `slug` en body:** El handler PATCH verifica `req.body.slug` ANTES de la validación Joi para poder devolver el mensaje exacto del spec. El flujo es:
1. `validatorHandler(idSchema, 'params')`
2. `authorize('admin')`
3. Handler: si `req.body.slug !== undefined` → `next(boom.badRequest('slug is immutable'))`
4. `validatorHandler(updateGroupSchema, 'body')` (no lo incluye, por lo que si llegó aquí sin `slug`, continúa)

Pero `validatorHandler` es un middleware, no puede ir después del handler inline. Se resuelve poniendo la verificación de `slug` dentro del handler async, antes de llamar al service:

```js
router.patch('/:id',
  validatorHandler(idSchema, 'params'),
  authorize('admin'),
  async (req, res, next) => {
    if (req.body.slug !== undefined) return next(boom.badRequest('slug is immutable'));
    const { error, value } = updateGroupSchema.validate(req.body, { abortEarly: false });
    if (error) return next(boom.badRequest(error.message));
    // ...
  }
);
```

Alternativa limpia: usar `validatorHandler` primero (que rechazará `slug` genéricamente), pero eso no da el mensaje exacto del spec. La solución es hacer la verificación de `slug` dentro del handler como primer paso, antes de la validación inline del body.

**Simplificación:** Para PATCH se hace la validación del body manualmente inline (como el spec muestra en `user.route.api.js` para el split de schemas). `updateGroupSchema` permite `unknown: false` (default en Joi) — si se pasa `slug`, Joi lanza error. Para dar el mensaje correcto, se hace la verificación de `slug` antes de llamar a Joi.

## Riesgos identificados (resueltos)

**R1 — Ciclo futuro `GroupService ↔ UserServices`**
Resolución: Documentar en `group.service.api.js` con comentario: "Depende de UserServices para sync de membresía. Si UserServices eventualmente necesita GroupService, extraer la lógica de sync a un MembershipService separado." El constructor injection (B3-D1) hace ese futuro refactor más limpio.

**R2 — `in` query con >30 slugs**
Resolución: Sin acción en v3. Límite documentado aquí: Firestore limita `in` a 30 valores. Un usuario en más de 30 grupos es inverosímil en v3. Guard defensiva diferida a v4.

**R3 — Gate admin/user en GET `/`**
Resolución: Confirmado: `if (req.user.role === 'admin')` en handler. Agregar comentarios inline `// admin: all groups` / `// user: own groups only`. Pasar `orderBy` del query string al `find()` para consistencia con admin path.

**R4 — Emails inexistentes en sync y camino a v4**
Resolución: **Fetch-first con error bloqueante.** Antes de cualquier write, hacer fetch de todos los usuarios del diff (`added` y `removed`). Si alguno lanza `boom.notFound`, el request falla con 400 (`boom.badRequest('User not found: email@example.com')`) — nada se escribe. El fetch de existencia es el mismo paso que v4 necesitará para obtener `docRef` y `groups` actual.

Camino a v4: batch writes requieren que todos los `DocumentReference` vengan de la misma instancia de Firestore. Actualmente `FireStoreAdapter` crea `new Firestore.Firestore()` por colección — instancias distintas. Para habilitar batch en v4:
  1. `[refactor]` Firestore singleton en `FireStoreAdapter`
  2. `[feat]` batch API expuesto en `FireStoreAdapter`
  3. `[feat]` `GroupService.update()` migra a batch — la lógica de fetch/diff/compute no cambia

**R5 — Atomicidad v3 / orden del sync**
Resolución: `super.update(group)` se llama **al final** del sync, después de todos los updates de usuarios. Iteración secuencial con `await` (fail-fast: falla en el primer write de usuario que falle). Loguear emails que fallen con `console.error()`. Comportamiento de inconsistencia parcial aceptado para v3.

**R6 — Bug: `users` ausente en PATCH borra miembros accidentalmente**
Resolución: **Bug a corregir en implementación.** El constructor `Group` debe tratar `users` ausente como `undefined` (no `[]`):
```js
this.users = users !== undefined ? users : undefined;
```
`cleanDocObject` elimina keys `undefined`, por lo que un PATCH sin `users` en el body no tocará el campo en Firestore. Un PATCH con `users: []` sí vaciará el grupo (correcto). Actualizar checkbox en el plan.
