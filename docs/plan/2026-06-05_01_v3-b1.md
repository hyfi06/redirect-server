# Plan: Bloque 1 — Autenticación (spec 2026-06-05_01_v3 §1.1–§1.6)

## Observaciones previas a la implementación

- `src/utils/auth/jwt.js` ya tiene código parcial pero con un typo: función `sing` en lugar de `sign`. El export también usa `sing`. Se corrige como parte de §1.1.
- `src/config/index.js` ya tiene `jwt: { jwtSecret, jwtTtl }`. El spec muestra `config.jwtSecret` (flat), pero el código existente usa `config.jwt.jwtSecret` (nested). **Decisión: mantener la estructura nested existente** — ya está en producción y cambiarla a flat requeriría tocar también `jwt.js`. Se usará `config.jwt.jwtSecret` y `config.jwt.jwtTtl` consistentemente.
- `.env.example` ya tiene `JWT_SECRET` y `JWT_TTL` — no requiere cambios.
- `google-oauth2.strategy.js` usa `passReqToCallback: true`, por lo que la firma del callback tiene `(request, accessToken, refreshToken, profile, done)` — se mantiene ese patrón al completar el callback.
- `User.auth` almacena `googleToken` y `googleRefreshToken` como campos dentro del objeto `auth` en Firestore. El `updateUserParser` ya maneja `auth` correctamente con dot-notation implícita (sobreescribe el objeto completo). Para el update de tokens en el callback de OAuth2 se construye el User con todos sus campos para que el parser no pierda otros campos de `auth`.

## Riesgos identificados

- El update de tokens en el callback de OAuth2 no bloquea el login (el spec lo indica explícitamente). Si `userService.update()` falla, el error se swallowea y el login continúa.
- `passport.authenticate('google', { failureRedirect })` en el callback devuelve JSON, no redirect — se usa callback custom para manejar el caso `user === false`.
- La estructura `config.jwt.*` ya funciona en el código actual de `jwt.js` — mantenerla evita romper tests existentes.
