## Objetivo

Sprint 2 cierra dos bloques independientes: mejoras internas en la capa de datos (§1.2 y §1.3) y enforcement de permisos en los handlers de redirect (§2.1, §2.2, §2.3, §2.4).

Al completarse Sprint 2, el repositorio debe tener:
- `FireStoreAdapter.update()` eliminando la lectura de existencia previa, confiando en el error gRPC NOT_FOUND.
- `GroupService.update()` sincronizando `User.groups` de forma atómica con un `WriteBatch`, sin estado inconsistente a mitad de loop.
- `PATCH /api/v1/redirects/:id` y `DELETE /api/v1/redirects/:id` evaluando las entradas `edit:{group}` y `delete:{group}` del array `permission`.
- `POST /api/v1/redirects` verificando que el grupo existe en Firestore antes de construir el path.
- Tests que cubran todas las nuevas ramas de acceso.

## Pasos

### §1.2 — Eliminar la doble lectura en update()

`FireStoreAdapter.update()` actualmente ejecuta un `docRef.get()` previo para verificar existencia, luego `docRef.update()`, y finalmente otro `docRef.get()` para retornar el documento actualizado. Esto supone 2 lecturas + 1 escritura por cada PATCH en redirects, users y groups. La spec establece que la lectura de existencia previa es redundante: Firestore lanza un error gRPC con código 5 (NOT_FOUND) cuando el documento no existe en `docRef.update()`. El segundo `get()` post-update se mantiene porque es necesario para que `docParser` pueda construir el modelo.

- [x] **Paso 1 — Modificar `FireStoreAdapter.update()` en `src/lib/firestore.js`.**

  Eliminar la lectura de existencia previa (`if (!(await docRef.get()).exists)`). Envolver la llamada `docRef.update()` en un bloque try/catch que intercepte el error gRPC. La spec confirma que el código de error es `5` y que la respuesta correcta es `boom.notFound('Resource not found')`. Relanzar cualquier otro error sin modificar. El `get()` posterior a la escritura se mantiene — es la única fuente de verdad para retornar el documento con todos sus campos tal como los ve el docParser.

  Criterio de aceptación: el método resulta en 1 lectura + 1 escritura en el caso exitoso, y en 1 escritura fallida en el caso NOT_FOUND, sin ninguna lectura previa.

- [x] **Paso 2 — Actualizar los tests de `FireStoreAdapter` en `src/lib/__test__/firestore.test.js`.**

  Los tests actuales pueden estar asertando la llamada a `get()` de existencia previa como parte del comportamiento esperado. El backend-engineer debe identificar qué mocks y aserciones cubren esa lectura antes de proceder.

  Los tests deben cubrir:
  - `update()` en documento existente: no llama a `get()` antes de `docRef.update()`; llama a `docRef.update()` con los datos correctos; llama a `get()` una vez al final; retorna el documento parseado.
  - `update()` en documento inexistente: `docRef.update()` lanza error con `code === 5`; `update()` lanza `boom.notFound`; no se llama a `get()`.
  - `update()` cuando `docRef.update()` lanza un error con código distinto de 5: el error se relanza sin transformar.

### §1.3 — Sync atómico de Group.users ↔ User.groups con batch write

`GroupService.update()` sincroniza `User.groups` mediante un loop secuencial de llamadas a `userService.update()`. Si el loop falla a mitad (por un error de red, Firestore throttle, o documento inexistente), el estado queda inconsistente: parte de los usuarios tienen `User.groups` actualizado y el resto no. Además, cada llamada a `userService.update()` pasa por `FireStoreAdapter.update()`, que después de §1.2 aún hace 1 lectura (el `get()` post-update) por usuario — lectura que es innecesaria en el contexto del sync porque el llamador no usa el valor retornado.

La solución es un `WriteBatch` de Firestore. El batch agrupa todas las operaciones (N actualizaciones de usuarios + 1 actualización del grupo) en una sola llamada atómica. O todas se aplican o ninguna.

Prerequisito: §1.1 completado (singleton de cliente Firestore en `src/lib/firestore-client.js`). El `WriteBatch` debe crearse desde el mismo cliente que referencia los `DocumentReference` incluidos en el batch.

- [x] **Paso 3 — Modificar `GroupService.update()` en `src/api/groups/services/group.service.api.js`.**

  Importar `firestoreClient` desde `src/lib/firestore-client.js`. Importar la clase `Timestamp` desde `@google-cloud/firestore` — es necesaria porque `WriteBatch.update()` no pasa por `FireStoreAdapter`, por lo tanto el auto-timestamping de `FireStoreAdapter.update()` no aplica y el campo `updated` debe construirse manualmente en cada entrada del batch.

  La lógica de fetch-first para construir el diff de usuarios (quién fue añadido y quién fue eliminado) se mantiene — alimenta el batch en lugar del loop. El batch debe incluir:
  - Una operación `batch.update()` por cada usuario añadido al grupo (actualiza `User.groups` añadiendo el nuevo slug).
  - Una operación `batch.update()` por cada usuario eliminado del grupo (actualiza `User.groups` removiendo el slug).
  - Una operación `batch.update()` para el grupo en sí (los campos del grupo que cambian).

  Todas las operaciones del batch deben incluir `updated: Timestamp.fromMillis(Date.now())` porque `WriteBatch.update()` acepta un `DocumentReference` y un objeto plano — no va por `FireStoreAdapter`.

  El backend-engineer debe confirmar cómo obtener los `DocumentReference` de cada usuario. El path de la colección `users` está hardcodeado en el adaptador; el batch necesita los mismos refs. La fuente correcta es `firestoreClient.collection('users').doc(user.id)`.

  Criterio de aceptación: si cualquier operación del batch falla, ninguna se aplica. Si el batch tiene éxito, todos los documentos (usuarios y grupo) quedan actualizados en Firestore en la misma operación lógica.

- [x] **Paso 4 — Actualizar los tests de `GroupService.update()` en `src/api/groups/services/__test__/group.service.api.test.js` (o donde vivan actualmente).**

  El backend-engineer debe localizar el archivo de test antes de proceder (`grep -rn "GroupService\|group.service" src/ --include="*.test.js"`).

  Los tests deben cubrir:
  - `update()` llama a `firestoreClient.batch()`.
  - El batch incluye una operación por cada usuario añadido, una por cada usuario eliminado, y una para el grupo.
  - `batch.commit()` es llamado exactamente una vez.
  - Si `batch.commit()` lanza un error, `update()` lo propaga.
  - No se llama a `userService.update()` directamente (la firma de las llamadas al servicio de usuario debe desaparecer del flujo).

### §2.1 — Enforcement de `edit:{group}` en PATCH /:id

`PATCH /api/v1/redirects/:id` actualmente permite el acceso solo al owner o a un admin. El campo `permission: string[]` del redirect puede contener entradas `"edit:{group}"` que fueron guardadas en Firestore pero nunca verificadas por el handler.

- [ ] **Paso 5 — Ampliar la verificación de acceso en el handler PATCH de `src/api/redirect/routes/redirect.route.api.js`.**

  El handler ya hace un fetch-first para obtener el documento antes de actualizar. Después de ese fetch, y antes de proceder con la actualización, añadir la verificación de acceso. La verificación debe evaluar si `req.user.role === 'admin'`, o si `existing.owner === req.user.email`, o si alguna entrada de `existing.permission` coincide con `edit:{group}` para algún grupo al que el usuario pertenezca según `req.user.groups`.

  La spec incluye la lógica acordada para esta verificación (§2.1 del spec). El backend-engineer debe implementarla de acuerdo con el patrón existente en `GET /:id`, donde la verificación de acceso es inline después del fetch.

  Si ninguna condición se cumple, el handler debe responder con 403 Forbidden usando `boom.forbidden`.

### §2.2 — Enforcement de `delete:{group}` en DELETE /:id

Análogo a §2.1 para el handler DELETE. Actualmente `DELETE /api/v1/redirects/:id` permite solo al owner o admin.

- [ ] **Paso 6 — Ampliar la verificación de acceso en el handler DELETE de `src/api/redirect/routes/redirect.route.api.js`.**

  El handler DELETE actualmente no tiene un fetch-first explícito — verificar si el documento es leído antes del delete. Si no lo es, añadir el fetch necesario para obtener `existing.owner` y `existing.permission` antes de ejecutar `redirectServiceApi.delete()`.

  La lógica de acceso sigue el mismo patrón que §2.1 pero evaluando entradas `"delete:{group}"` en lugar de `"edit:{group}"`. La spec incluye la lógica acordada (§2.2 del spec). Si ninguna condición se cumple, responder con 403.

  Nota: Paso 5 y Paso 6 afectan el mismo archivo y pueden implementarse en el mismo ciclo. Irán en el mismo commit `[feat]`.

### §2.3 — Verificación de existencia del grupo en POST /:id

`POST /api/v1/redirects` verifica que el usuario pertenece al grupo consultando `req.user.groups` del JWT, pero no verifica que el documento del grupo exista en la colección `groups` de Firestore. Un usuario podría crear paths bajo un slug de grupo que fue eliminado de la base de datos.

- [ ] **Paso 7 — Instanciar `GroupService` dentro del redirect router en `src/api/redirect/routes/redirect.route.api.js`.**

  La decisión adoptada en §2.3 del spec es instanciar `GroupService` dentro del router — igual que el patrón actual donde cada router es el composition root de sus dependencias. No se pasa como parámetro al router.

  El `GroupService` requiere a su vez un `UserService` inyectado en su constructor (ver arquitectura en CLAUDE.md). El backend-engineer debe revisar cómo `GroupService` es instanciado en `src/api/groups/routes/group.route.api.js` y replicar el mismo patrón.

- [ ] **Paso 8 — Añadir la verificación de existencia del grupo en el handler POST de `src/api/redirect/routes/redirect.route.api.js`.**

  La verificación aplica solo para usuarios no-admin. Los admin pueden crear paths en cualquier namespace (ver reglas de path en CLAUDE.md — §Path namespace rules). Para usuarios regulares, antes de construir el `fullPath`, llamar a `GroupService.getBySlug(group)` donde `group` es el primer segmento del path proporcionado por el usuario. Si `getBySlug` lanza `boom.notFound`, propagar el error con un mensaje que indique que el grupo no existe.

  El backend-engineer debe verificar qué lanza `getBySlug` cuando no encuentra el slug — si lanza `boom.notFound` directamente o si retorna `null`. La respuesta determina si se propaga o se construye un boom explícito.

## Orden de implementación recomendado

**Orden:** §1.2 → §1.3 → §2.1 + §2.2 → §2.3 → §2.4

**Razones:**

1. **§1.2 primero** porque es el cambio de menor superficie: un método en un archivo, tests en un archivo de test ya existente, sin nuevas dependencias. Elimina una lectura por cada PATCH en las tres colecciones. Completarlo primero permite verificar que el singleton de §1.1 no introdujo regresiones antes de construir §1.3 sobre él.

2. **§1.3 segundo** porque depende de §1.1 (completado en Sprint 1) y es el cambio más complejo del sprint. Hacerlo después de §1.2 garantiza que la capa `FireStoreAdapter` ya está estabilizada. El batch write cambia la firma de interacción con Firestore de forma significativa — conviene que sea el segundo cambio en verde antes de tocar los handlers de redirect.

3. **§2.1 y §2.2 juntos** porque son análogos y afectan el mismo archivo (`redirect.route.api.js`) en el mismo patrón de verificación de acceso post-fetch. El handler PATCH y DELETE comparten la lógica de `owner || admin || group-permission` con solo el scope cambiando. Implementarlos en el mismo ciclo evita duplicar la revisión del archivo y produce un commit `[feat]` y un `[test]` más cohesivos.

4. **§2.3 después de §2.1/§2.2** porque añade una dependencia nueva al redirect router (`GroupService`), que requiere entender cómo está organizado el router en ese momento. Es mejor introducir la nueva dependencia después de que las verificaciones de acceso del Paso 5 y 6 ya estén integradas.

5. **§2.4 cubierto por los commits de test de cada unidad** — los Pasos 9-11 se implementan como parte de los ciclos `[feat] → [test]` de §2.1+§2.2 y §2.3, no en un commit separado final.

## Notas para el backend-engineer

**§1.2 — El error gRPC tiene campo `code`, no `status`:**

La spec establece que el error NOT_FOUND tiene `code === 5`. El backend-engineer debe verificar la forma exacta del objeto de error que lanza `docRef.update()` inspeccionando el SDK de `@google-cloud/firestore` o los tipos de error de `@grpc/grpc-js`. La propiedad `code` (numérica) es la que diferencia NOT_FOUND de otros errores. No usar `err.message` para este chequeo — los mensajes de error del SDK pueden cambiar entre versiones.

**§1.3 — `WriteBatch.update()` no pasa por `FireStoreAdapter`:**

El batch recibe `DocumentReference` y un plain object. No hay `docParser`, no hay `updateParser`, y no hay auto-timestamping. El backend-engineer debe añadir `updated: Timestamp.fromMillis(Date.now())` manualmente en cada entrada del batch — tanto para los usuarios como para el grupo. Omitirlo dejará los documentos sin el campo `updated` actualizado, lo que afecta el ordenamiento por defecto de `CrudService.find()` (que ordena por `updated desc`).

**§1.3 — Colección `users` hardcodeada en el batch:**

`WriteBatch.update()` necesita el `DocumentReference` completo. Para los usuarios, la referencia es `firestoreClient.collection('users').doc(user.id)`. El nombre de la colección `'users'` está implícito en la instanciación de `FireStoreAdapter` para la colección de usuarios. El backend-engineer debe verificar el nombre exacto de la colección revisando cómo `UserService` instancia su `FireStoreAdapter`.

**§2.1 + §2.2 — Verificar si DELETE ya tiene fetch-first:**

Antes de implementar §2.2, el backend-engineer debe leer el handler DELETE actual en `src/api/redirect/routes/redirect.route.api.js` para determinar si el documento es leído antes del delete. La verificación de acceso requiere los campos `owner` y `permission` del documento existente. Si el handler no tiene fetch-first, añadirlo es parte del Paso 6. No es aceptable verificar el acceso después de borrar el documento.

**§2.3 — Patrón de instanciación de `GroupService`:**

`GroupService` recibe `UserService` en su constructor (D12 en la arquitectura — ver CLAUDE.md). El backend-engineer debe leer cómo se instancia en `src/api/groups/routes/group.route.api.js` y replicar el mismo patrón en `src/api/redirect/routes/redirect.route.api.js`. No instanciar `GroupService` sin inyectarle el `UserService` — el constructor lo requiere para el sync de §1.3.
