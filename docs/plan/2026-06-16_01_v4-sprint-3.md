# Plan — Sprint 3: API Keys

**Spec:** docs/spec/2026-06-10_01_v4.md §3.1 – §3.7
**Rama:** dev
**Fecha:** 2026-06-16

---

## Objetivo

Sprint 3 implementa el sistema completo de API Keys para 1kg.me. Al completarse, cualquier usuario autenticado podrá crear tokens Bearer con scopes limitados (`sk_1kg_…`) que otorgan acceso programático a los endpoints de redirects — sin exponer las credenciales OAuth2/JWT de la sesión interactiva.

Al completarse Sprint 3, el repositorio debe tener:

- Subcolección `users/{userId}/apiKeys/{keyId}` en Firestore con el schema acordado en §3.1.
- Índice `collectionGroup('apiKeys').where('keyHash')` en `firestore.indexes.json` (§3.7).
- `ApiKey` model, `api-key.parser.js`, `ApiKeyService`, y `api-key.schema.js` como la capa de dominio del recurso (§3.5).
- Tres endpoints bajo `/api/v1/users/me/api-keys`: `GET /`, `POST /`, `DELETE /:keyId` (§3.3).
- `authenticate` middleware ampliado: detecta el prefijo `sk_1kg_`, resuelve el hash, carga el usuario, inyecta `req.user.apiKey` (§3.6 + §3.2).
- Cache de lookup por `keyHash` en `node-cache`, TTL 30s; invalidado al revocar (§3.2).
- Middleware `authorizeApiKeyScope(requiredScope)` aplicado por ruta en el router de redirects (§3.4).
- Rechazo 403 en los routers de users y groups cuando `req.user.apiKey` está definido (§3.4).
- Tests que cubran todas las ramas nuevas en cada unidad.

## Pasos

### §3.7 — Índice Firestore para lookup de API Keys

El flujo de autenticación (§3.2) hace `collectionGroup('apiKeys').where('keyHash', '==', keyHash).limit(1)`. Esta es una query sobre un collection group — Firestore la rechaza con `FAILED_PRECONDITION` si no existe el índice. El índice debe estar en producción antes de que cualquier request con API Key llegue a ese código.

- [x] **Paso 1 — Añadir el índice `apiKeys/keyHash` a `firestore.indexes.json`.**

  Añadir una entrada al array `indexes` de `firestore.indexes.json` con:
  - `collectionGroup`: `"apiKeys"`
  - `queryScope`: `"COLLECTION_GROUP"`
  - `fields`: un solo campo `keyHash` con `order: "ASCENDING"`

  El archivo ya tiene tres índices para `redirects` y `groups`. El nuevo índice se añade a continuación. No modificar los índices existentes.

  Criterio de aceptación: `npm run indexes` (que ejecuta `node scripts/manage-indexes.js`) crea el índice en el proyecto GCP activo sin errores. El índice queda en estado `CREATING` o `READY`.

  Tipo de commit: `[chore]`

### §3.5 — ApiKey: model, parser, service, schema

Esta es la capa de dominio del recurso. Todo lo que viene después depende de ella. `ApiKeyService` no extiende `CrudService` porque la ruta de la subcolección incluye el `userId` como segmento dinámico — `CrudService` espera un nombre de colección fijo en el constructor. `ApiKeyService` accede directamente al cliente Firestore singleton (`src/lib/firestore-client.js`).

- [x] **Paso 2 — Crear `src/api/users/models/api-key.model.js`.**

  La clase `ApiKey` recibe todos los campos del schema de §3.1 y los asigna en el constructor. El campo `active` tiene default `true` cuando no se proporciona. El campo `expiresAt` tiene default `null`. `keyHash` nunca se expone en `toPublic()` — el método solo retorna `id`, `name`, `prefix`, `scopes`, `expiresAt`, `createdAt`, `lastUsedAt`, y `active`. El modelo no tiene lógica de negocio — solo estructura y serialización pública. Ver el shape completo del constructor en §3.5 del spec.

- [x] **Paso 3 — Crear `src/api/users/parsers/api-key.parser.js`.**

  Tres funciones siguiendo el patrón establecido en `src/api/users/parsers/user.parser.js`:

  - `apiKeyParser(docSnap)`: lee el `DocumentSnapshot`, extrae `docSnap.ref.id`, convierte `createdAt` y `lastUsedAt` (ambos Timestamps, `lastUsedAt` puede ser null) y `expiresAt` (Timestamp o null) a `Date` o `null`. Retorna `new ApiKey({...})`.

  - `createApiKeyParser(apiKey)`: toma un `ApiKey` y retorna el plain object para escribir en Firestore. Incluye `keyHash`, `prefix`, `name`, `scopes`, `expiresAt` (Timestamp o null), `active: true`, `lastUsedAt: null`. No incluye `id`. El campo `createdAt` lo pone `FireStoreAdapter.create()` automáticamente como Timestamp... excepto que `ApiKeyService` no usará `FireStoreAdapter`. El backend-engineer debe verificar si `FireStoreAdapter.create()` puede ser reutilizado para subcollections o si el Timestamp `createdAt` debe construirse manualmente en el parser o en el service. El campo equivalente a `created`/`updated` en el resto de recursos es aquí `createdAt` — no hay `updated` en `apiKey` porque las keys son inmutables post-creación.

  - `updateApiKeyParser(apiKey)`: solo necesita revocar una key (`active = false`). El plain object de revocación contiene únicamente `{ active: false }`. El backend-engineer puede decidir si el updateParser recibe un `ApiKey` completo o si la revocación es suficientemente simple como para prescindir del parser en el handler directo — lo que sea más coherente con el patrón establecido.

- [x] **Paso 4 — Crear `src/api/users/services/api-key.service.js`.**

  `ApiKeyService` recibe `userId` en cada método — la subcolección depende de él. El backend-engineer debe decidir si `userId` se pasa al constructor (un servicio por usuario, instanciado en el handler con `req.user.userId`) o como parámetro de cada método (un singleton compartido). El patrón más simple y coherente con el resto de servicios es pasarlo a cada método, evitando instancias por-request.

  Métodos requeridos:

  - `list(userId)`: retorna todas las `apiKey` de la subcolección `users/{userId}/apiKeys`, ordenadas por `createdAt` descendente. Aplica `apiKeyParser` a cada doc.

  - `create(userId, apiKey)`: escribe el documento en `users/{userId}/apiKeys`. El `id` lo asigna Firestore automáticamente. Verifica antes de escribir que el usuario no supera el límite de 10 keys activas — si el count de docs con `active === true` es 10 o más, lanza `boom.badRequest('API Key limit reached')`. Retorna el `ApiKey` parseado. El token plano (`sk_1kg_…`) que se devuelve al cliente nunca pasa por el servicio ni por Firestore — se genera en el handler, se hashea con SHA-256, y solo el hash llega al servicio.

  - `revoke(userId, keyId)`: actualiza `active = false` en `users/{userId}/apiKeys/{keyId}`. Lanza `boom.notFound` si el doc no existe.

  - `findByHash(keyHash)`: query `collectionGroup('apiKeys').where('keyHash', '==', keyHash).limit(1)`. Retorna `{ apiKey, userId }` donde `userId` se extrae del path del documento (`docSnap.ref.parent.parent.id`). Retorna `null` si no encuentra resultados — el caller (authenticate) convierte esto en 401. Este método es el único que usa `collectionGroup` en lugar de una subcollection por userId; el backend-engineer debe importar `firestoreClient` directamente para construir la query.

- [x] **Paso 5 — Crear `src/api/users/schemas/api-key.schema.js`.**

  Schemas Joi para los tres endpoints:

  - `listApiKeysSchema`: no tiene params ni query — vacío o no definido; el validatorHandler puede omitirse en `GET /me/api-keys`.

  - `createApiKeySchema` (body): `name` string requerido, max 100 chars. `scopes` array de strings requerido, al menos 1 elemento, cuyos valores son un enum del catálogo exacto: `read:redirects`, `write:redirects`, `read:users`, `write:users`, `read:groups`, `write:groups`. `expiresAt` fecha ISO opcional, puede ser null.

  - `deleteApiKeySchema` (params): `keyId` string requerido.

  La validación de que un usuario regular no puede pedir scopes de admin (`read:users`, etc.) no es responsabilidad del schema Joi — la regla involucra el `req.user.role`, no solo el body. Pertenece al handler.

- [x] **Paso 6 — Escribir tests para `ApiKey` model, parsers, y `ApiKeyService`.**

  Tests en `src/api/users/__test__/` (o en subdirectorios `__test__/` al lado de cada archivo, siguiendo el patrón del proyecto):

  - `api-key.model.js`: constructor con defaults, `toPublic()` nunca expone `keyHash`.
  - `api-key.parser.js`: `apiKeyParser` convierte Timestamps a Date, trata `null` en `lastUsedAt`/`expiresAt`. `createApiKeyParser` incluye `keyHash` y excluye `id`. `updateApiKeyParser` retorna `{ active: false }`.
  - `api-key.service.js`: mockear `firestoreClient`. `list()` retorna docs parseados. `create()` rechaza con 400 si hay 10 keys activas; escribe el doc si hay menos. `revoke()` actualiza el doc; lanza notFound si no existe. `findByHash()` extrae `userId` del path; retorna null si no hay resultados.

  Tipo de commit: `[feat]` para Pasos 2-5, `[test]` para Paso 6.

### §3.3 — Endpoints CRUD de API Keys

Los tres endpoints se montan en un sub-router que se registra dentro de `user.route.api.js`. El sub-router hereda el middleware `authenticate` del router padre porque se monta después del `userRouterApi.use(authenticate)` en la línea 20 de `src/api/users/routes/user.route.api.js`. El backend-engineer debe verificar que el orden de registro garantiza que `authenticate` se ejecuta antes que los handlers del sub-router.

**Nota de montaje importante:** `GET /me` en el router de usuarios está declarado antes de `GET /:id` para evitar que Express interprete `"me"` como un id. Los endpoints de API Keys se montan bajo `/me/api-keys` — el backend-engineer debe verificar que el sub-router se registra antes de `GET /:id` o, como alternativa, que el path `/me/api-keys` es lo suficientemente específico para no colisionar con `/:id`. La forma segura es registrar el sub-router inmediatamente después de `GET /me`, antes de `GET /:id`.

- [x] **Paso 7 — Crear `src/api/users/routes/api-key.route.js`.**

  Sub-router con tres rutas:

  - `GET /`: llama a `apiKeyService.list(req.user.userId)`. Retorna 200 con `{ message: 'api keys retrieved', data: keys.map(k => k.toPublic()) }`.

  - `POST /`: valida el body con `createApiKeySchema`. Verifica que si `req.user.role !== 'admin'`, los `scopes` del body no contienen scopes de admin (`read:users`, `write:users`, `read:groups`, `write:groups`); si contienen, responde 403. Genera el token: string aleatorio con prefijo `sk_1kg_` (32 chars base62 aleatorios). Computa `keyHash = SHA-256(token)` en hex. Extrae `prefix` como los primeros 8 chars del token. Construye `new ApiKey({ name, keyHash, prefix, scopes, expiresAt })` y llama a `apiKeyService.create(req.user.userId, apiKey)`. Retorna 201 con `{ message: 'api key created', data: { ...key.toPublic(), token } }` — `token` solo aparece aquí.

  - `DELETE /:keyId`: valida params con `deleteApiKeySchema`. Llama a `apiKeyService.revoke(req.user.userId, keyId)`. Invalida la cache: elimina la entrada en `nodeCache` cuya key es el `keyHash` del doc. El backend-engineer debe obtener el `keyHash` antes de revocar (llamar a `apiKeyService.findByHash` no es útil aquí — ya se tiene el `keyId`; el servicio debe exponer un método `findOne(userId, keyId)` o el handler puede obtener el doc directamente). Discusión: la forma más simple es que `apiKeyService.revoke()` retorne el `keyHash` del documento antes de marcarlo como inactivo, para que el handler lo use para invalidar la cache. El backend-engineer elige la implementación más limpia.

  Retorna 200 con `{ message: 'api key revoked', data: { id: keyId } }`.

- [x] **Paso 8 — Montar el sub-router en `src/api/users/routes/user.route.api.js`.**

  Importar `apiKeyRouterApi` del archivo del Paso 7. Registrar con `userRouterApi.use('/me/api-keys', apiKeyRouterApi)` inmediatamente después del handler `GET /me` (línea ~30) y antes del handler `GET /:id`. Instanciar `ApiKeyService` en `user.route.api.js` y pasarlo al sub-router, o instanciarlo directamente en `api-key.route.js` como singleton al nivel de módulo (patrón actual del proyecto — cada router crea sus propias dependencias).

- [x] **Paso 9 — Escribir tests para los tres endpoints.**

  Tests en `src/api/users/routes/__test__/api-key.route.test.js` (crear el directorio `__test__/` si no existe).

  Tests requeridos:

  - `POST /me/api-keys` — caso exitoso: body válido, scope de usuario, retorna 201 con `token` en el body.
  - `POST /me/api-keys` — límite alcanzado: service lanza 400 cuando hay 10 keys activas.

  - `POST /me/api-keys` — scope de admin por admin: retorna 201.

### §3.6 + §3.2 — Middleware `authenticate` ampliado para API Keys

Este paso convierte `authenticate` de un verificador de JWT a un dispatcher que delega a la rama JWT o a la rama API Key según el prefijo del token.

- [x] **Paso 10 — Refactorizar `src/middleware/authenticate.middleware.js`.**

  La función `authenticate` actual extrae el token del header y lo pasa directamente a `verify()`. El refactor la convierte en un dispatcher:

  - Extrae el token del header `Authorization: Bearer …`. Si no existe, responde 401.
  - Si el token comienza con `sk_1kg_`, llama a la función interna `authenticateApiKey(token, req, next)`.
  - En caso contrario, llama a la función interna `authenticateJwt(token, req, next)` que contiene el comportamiento actual (envolver `verify()` en try/catch).

  La función `authenticateApiKey(token, req, next)` implementa el flujo del §3.2:

  1. Computa `keyHash = SHA-256(token)` en hex. Usar el módulo `crypto` de Node.js nativo — no instalar dependencias externas.
  2. Consulta la cache: `nodeCache.get(keyHash)`. Si hay hit, el valor cacheado es el objeto `{ apiKey, user }` ya construido — asignar directamente a `req.user` y llamar `next()`.
  3. Si no hay hit: llama a `apiKeyService.findByHash(keyHash)`. Si retorna `null`, responde 401.
  4. Verifica `keyDoc.active === true`. Si no, responde 401.
  5. Verifica `keyDoc.expiresAt === null || keyDoc.expiresAt > new Date()`. Si expiró, responde 401.
  6. Carga el usuario: `userService.findOne(userId)`.
  7. Construye `req.user = { userId: user.id, email: user.email, role: user.role, groups: user.groups, apiKey: { id: keyDoc.id, scopes: keyDoc.scopes } }`.
  8. Guarda en cache: `nodeCache.set(keyHash, req.user, 30)` (TTL 30 segundos).
  9. Llama `next()`.

  `ApiKeyService` y `UserServices` deben instanciarse como singletons en el módulo `authenticate.middleware.js`, igual que los demás servicios se instancian en sus respectivos routers. Alternativa: recibirlos como parámetros para facilitar el testing — el backend-engineer elige. La instanciación directa es más simple y consistente con el patrón del proyecto; la inyección facilita el mock en tests sin necesitar jest module mocking.

- [x] **Paso 11 — Escribir tests para `authenticate` con API Key.**

  Tests en `src/middleware/__test__/authenticate.middleware.test.js` (el archivo puede ya existir para los tests JWT — extenderlo o crear uno nuevo siguiendo el patrón del directorio).

  Tests requeridos:

  - Token sin prefijo `sk_1kg_`: flujo JWT (comportamiento actual no debe regresionar).
  - Token `sk_1kg_…` con cache hit: no llama a Firestore; asigna `req.user` desde cache; `req.user.apiKey` está definido.
  - Token `sk_1kg_…` sin cache, key activa y vigente: hace lookup en Firestore; construye `req.user` con `apiKey`; guarda en cache.
  - Token `sk_1kg_…`, key no encontrada: 401.
  - Token `sk_1kg_…`, key con `active === false`: 401.
  - Token `sk_1kg_…`, key con `expiresAt` en el pasado: 401.
  - Sin header `Authorization`: 401.
  - `req.user.apiKey` tiene `id` y `scopes` con los valores correctos del doc Firestore.

  Tipo de commit: `[feat]` para Paso 10, `[test]` para Paso 11.

### §3.4 — Enforcement de scope y rechazo en users/groups

Con `req.user.apiKey` ya disponible (tras §3.6), este paso añade las dos capas de enforcement: scope por ruta en redirects, y rechazo total en users/groups.

- [x] **Paso 12 — Crear `src/middleware/authorize-api-key-scope.middleware.js`.**

  Middleware factory `authorizeApiKeyScope(requiredScope)` que retorna un handler Express. Lógica:

  - Si `req.user.apiKey` es `undefined` (autenticado con JWT): llama `next()` sin verificar nada. El usuario JWT tiene acceso pleno a las rutas que pase el middleware `authorize`.
  - Si `req.user.apiKey` existe y `req.user.apiKey.scopes` incluye `requiredScope`: llama `next()`.
  - Si `req.user.apiKey` existe y `requiredScope` no está en `scopes`: responde 403 con `boom.forbidden('API Key scope required: ' + requiredScope)`.

- [x] **Paso 13 — Aplicar `authorizeApiKeyScope` en `src/api/redirect/routes/redirect.route.api.js`.**

  Importar `authorizeApiKeyScope`. Añadir el middleware a cada ruta del redirect router, encadenado después de `authenticate` (ya aplicado a nivel de router) y antes del handler:

  - `GET /`: `authorizeApiKeyScope('read:redirects')`
  - `GET /:id`: `authorizeApiKeyScope('read:redirects')`
  - `POST /`: `authorizeApiKeyScope('write:redirects')`
  - `PATCH /:id`: `authorizeApiKeyScope('write:redirects')`
  - `DELETE /:id`: `authorizeApiKeyScope('write:redirects')`

  El middleware es no-op para requests JWT — los tests existentes no deben regresionar.

- [x] **Paso 14 — Añadir rechazo de API Keys en `src/api/users/routes/user.route.api.js` y `src/api/groups/routes/group.route.api.js`.**

  En ambos routers, después de `authenticate` y antes de cualquier handler, añadir un middleware inline que verifique `req.user.apiKey === undefined`. Si `req.user.apiKey` está definido, responder 403 con `boom.forbidden('API Keys cannot be used on this resource')`.

  Excepción acordada en el spec: la implementación simple rechaza todas las API Keys en users/groups, sin excepción para admins con scopes de admin. Esta es la política de v4 — si en el futuro se decide habilitar acceso admin via API Key, se añade la excepción entonces.

  El middleware de rechazo debe registrarse en el router de nivel correcto para que aplique a todas las rutas. El backend-engineer debe verificar si se registra con `router.use()` (afecta todas las rutas del router) o inline en cada ruta. `router.use()` después de `authenticate` es la forma más simple.

  - `POST /api/v1/redirects` con API Key sin `write:redirects`: responde 403.

## Orden de implementación recomendado

**Orden:** §3.7 → §3.5 → §3.3 → §3.6+§3.2 → §3.4

**Razones:**

1. **§3.7 primero** porque es una entrada JSON en un archivo de configuración, no requiere código, y habilita que las queries `collectionGroup('apiKeys').where('keyHash')` funcionen en producción. Es el cambio de menor riesgo y de mayor urgencia para el entorno real: sin el índice, cualquier request con API Key fallará en producción con un error de Firestore, independientemente del estado del código. Hacerlo primero también permite que el índice quede en estado `CREATING` mientras el resto del sprint se implementa — Firestore puede tardar minutos en construirlo.

2. **§3.5 segundo** porque es la fundación: el model, los parsers, el service, y el schema son importados por los Pasos 7-14. Sin `ApiKey` y `ApiKeyService` definidos, ningún otro paso puede completarse. Implementar model → parsers → service → schema en secuencia, con tests después de los cuatro archivos.

3. **§3.3 tercero** porque los endpoints son la superficie visible del sprint — validan que model + parsers + service funcionan de extremo a extremo. Con §3.5 en verde, los handlers de `GET /me/api-keys`, `POST /me/api-keys`, y `DELETE /me/api-keys/:keyId` pueden implementarse y testearse con el service real. El montaje en `user.route.api.js` (Paso 8) es el único punto de integración con el código existente y es no-breaking.

4. **§3.6 + §3.2 cuarto** porque depende de `ApiKeyService` (para `findByHash`) y de `UserServices` (para cargar el usuario). Con §3.5 completo, el refactor de `authenticate` puede implementarse. Este paso toca el middleware más transversal del sistema — hacerlo después de §3.3 garantiza que los endpoints de API Keys ya funcionan con autenticación JWT antes de introducir el segundo path de autenticación.

5. **§3.4 último** porque el middleware `authorizeApiKeyScope` necesita que `req.user.apiKey` ya exista (producido por §3.6). Y el rechazo en users/groups solo tiene sentido verificarlo cuando los dos primeros paths de autenticación están completos. §3.4 es también el paso con mayor superficie de tests — conviene que sea el último, cuando el resto del sistema está estabilizado.

## Notas para el backend-engineer

**Generación del token — módulo `crypto` nativo:**

El token `sk_1kg_{32 chars base62}` debe generarse con el módulo `crypto` de Node.js nativo. No instalar `uuid` ni ninguna dependencia externa para esto. `crypto.randomBytes(N)` produce bytes aleatorios seguros; convertirlos a base62 requiere un encoder manual (alphabet `0-9A-Za-z`). El SHA-256 del token se computa con `crypto.createHash('sha256').update(token).digest('hex')`.

**`ApiKeyService` no extiende `CrudService`:**

`CrudService` recibe un nombre de colección fijo en el constructor y construye su `FireStoreAdapter` con ese nombre. La subcolección `apiKeys` vive bajo `users/{userId}` — el path cambia con cada usuario. `ApiKeyService` debe importar `firestoreClient` directamente desde `src/lib/firestore-client.js` y construir las referencias con `firestoreClient.collection('users').doc(userId).collection('apiKeys')`. No hay `FireStoreAdapter` de por medio en este servicio.

**Timestamps en `ApiKeyService` — sin auto-timestamping:**

Dado que `ApiKeyService` no usa `FireStoreAdapter`, no hay auto-timestamping. Los campos `createdAt` y `lastUsedAt` deben construirse manualmente como `Timestamp.fromMillis(Date.now())` (importar `Timestamp` de `@google-cloud/firestore`). `lastUsedAt` se inicializa como `null` en creación. `expiresAt` si el cliente envía una fecha ISO, convertir a Timestamp antes de escribir en Firestore.

**Cache invalidation en `DELETE /me/api-keys/:keyId`:**

Al revocar una key, el handler debe eliminar la entrada `nodeCache.del(keyHash)` con el hash de la key revocada. El problema es que el handler tiene el `keyId` del path param pero no el `keyHash` directamente. La solución recomendada: el método `apiKeyService.revoke(userId, keyId)` retorna el documento antes de marcarlo como inactivo, para que el handler tenga el `keyHash`. Alternativa: el handler hace un `findOne(userId, keyId)` antes del revoke. El backend-engineer elige la opción más limpia, documentando la decisión con un comentario inline.

**Extracción de `userId` desde el path del documento en `findByHash`:**

`collectionGroup('apiKeys')` retorna documentos cuyo path es `users/{userId}/apiKeys/{keyId}`. Para extraer el `userId`: `docSnap.ref.parent.parent.id`. `docSnap.ref` es la referencia al doc `apiKey`. `.parent` es la `CollectionReference` de `apiKeys`. `.parent` nuevamente es la `DocumentReference` del usuario. `.id` es el `userId`. El backend-engineer debe verificar este chain en el Firestore SDK antes de asumir que es correcto.

**Montaje del sub-router — orden de declaración:**

El orden de declaración en Express importa. `GET /me` debe declararse antes de `GET /:id`. El sub-router de API Keys se monta en `/me/api-keys` — este path comienza con `/me`, por lo que Express lo resolverá correctamente si el sub-router se registra con `userRouterApi.use('/me/api-keys', apiKeyRouterApi)`. Sin embargo, el backend-engineer debe verificar que este `use()` no colisiona con el handler `GET /me` ya registrado. La ruta `/me/api-keys` es más específica que `/me` — Express los distingue correctamente cuando ambos son declaraciones separadas.

**Rechazo de API Keys en users/groups — no usar `authorizeApiKeyScope`:**

El rechazo en users/groups no es un problema de scope faltante — es un rechazo total del mecanismo de autenticación. La semántica es diferente: `authorizeApiKeyScope` dice "tienes el scope correcto?"; el rechazo en users/groups dice "este mecanismo de auth no está permitido aquí, independientemente de los scopes". Usar `router.use()` con un middleware dedicado o inline, no encadenar `authorizeApiKeyScope`.

**Tests de §3.6 — mockear dependencias de `authenticate`:**

`authenticate` después del refactor instancia (o recibe) `ApiKeyService` y `UserServices`. Para testear el path de API Key sin Firestore real, el backend-engineer debe poder inyectar mocks de estos servicios. Si se instancian como singletons dentro del módulo `authenticate.middleware.js`, mockearlos requiere `jest.mock()` del módulo del servicio. Si se pasan como parámetros (factory function o closure), el test los inyecta directamente. El backend-engineer debe elegir la estrategia que resulte en tests más claros y que sea coherente con cómo se testean los otros middlewares del proyecto.
