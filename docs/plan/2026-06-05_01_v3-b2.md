# Plan: Bloque 2 — API de redirects protegida (spec 2026-06-05_01_v3 §2.1–§2.3)

## Observaciones previas a la implementación

- `authenticate` y `authorize` ya existen y están listos en `src/middleware/`.
- `redirect.route.api.js` actualmente usa `owner` y `group` tomados de `req.query` en GET — esos campos desaparecen del schema y pasan a obtenerse de `req.user`.
- `createRedirectSchema` tiene `owner` como requerido — se elimina y se añade `group` opcional.
- `updateRedirectSchema` tiene `owner` opcional — se elimina.
- `getRedirectQuerySchema` tiene `owner` y `group` requeridos — se simplifican a solo `orderBy`, `offset`, `limit`.
- El spec (D5) indica que `fullPath = group ? \`${group}/${path}\` : path`. La validación del path en Joi (`uri relativeOnly`) debe funcionar con tanto `"/foo"` como `"/foo/bar"` — el campo `path` del body es solo el segmento sin el grupo.
- La decisión D10 confirma el uso de `array-contains-any` dentro de `Filter.or`. La versión del SDK es `^7.11` — mayor que `7.1`, por lo que el fallback no es necesario.
- El handler de GET actual fallará si `groups` está vacío (el `Filter.or` con `array-contains-any` requiere array no vacío). El spec lo contempla: se usa solo `Filter.where('owner', '==', email)` cuando `readPermissions.length === 0`.
- PATCH y DELETE actualmente no verifican ownership. Se añade la verificación antes de la operación.
- `redirectServicieApi.findOne(id)` ya existe (heredado de `CrudService`) — se usa en los handlers de PATCH y DELETE.

## Riesgos identificados

- El campo `group` en `createRedirectSchema` usa `.lowercase().pattern(/^[a-z0-9-]+$/)` según el spec. El `path` del body es un URI relativo (`/foo` o `/foo/bar`) — el handler concatena `${group}/${path}` pero `path` empieza con `/`, por lo que `fullPath` quedaría `"fc//foo"`. Se debe decidir si `path` se normaliza quitando el `/` inicial antes de concatenar, o si el schema debe aceptar paths sin `/` inicial. **Decisión requerida antes de implementar POST.**

  Analizando el spec: `Joi.string().uri({ allowRelative: true, relativeOnly: true })` para `path` en `createRedirectSchema`. El spec muestra en D5: `fullPath = group ? \`${group}/${path}\` : path`. Si `path = "/seminar"`, el resultado sería `"fc//seminar"`. Si `path = "seminar"`, el resultado sería `"fc/seminar"`. 
  
  Revisando `getByPath` en el servicio: compara `path` en Firestore con el `path` del request. El catch-all de redirects usa `req.path` que siempre empieza con `/`. Esto sugiere que el `path` almacenado en Firestore empieza con `/`.

  **Conclusión:** El `path` del body en el POST debe permitir paths relativos con o sin `/` inicial. Al construir `fullPath`:
  - Si `group` está presente: `fullPath = \`/${group}/${path.replace(/^\//, '')}\``
  - Si no hay `group` (admin): `fullPath = path` (ya tiene `/`)
  
  Esto es un comportamiento no cubierto explícitamente por el spec — se reporta al usuario para confirmación antes de implementar.
