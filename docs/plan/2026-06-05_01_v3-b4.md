## Observaciones previas a la implementación

### Estado actual

- `authenticate` y `authorize` ya existen y están completos (`src/middleware/authenticate.middleware.js`, `src/middleware/authorize.middleware.js`).
- `userRouterApi` no tiene ningún middleware de autenticación — todas las rutas son públicas.
- `updateUserSchema` en `user.schema.js` es único y permite `groups` y `auth`, pero no `role`. No distingue entre admin y usuario regular.
- El `User` model vive en `src/api/users/models/user.js` (sin sufijo `.api.`), a diferencia del patrón `*.model.api.js` que describe el stack overview. No es un problema — es el único modelo bajo `users/`.
- `UserServices` no tiene método `findOne` propio — hereda `findOne(id)` de `CrudService`, que llama `db.get(id)` → lanza `boom.notFound` si no existe. Correcto para `GET /me`.
- `src/api/index.js` ya monta `groupRouterApi` y `authRouterApi` — evidencia de que los Bloques 1 y 3 están implementados.
- No existe un archivo de schema separado para el query de `GET /api/v1/users` (por ejemplo, `getUsersQuerySchema`). El handler actual usa `req.query` sin validación Joi. Está fuera del alcance del Bloque 4 — no se toca.
- `CrudService.find(query, options)` recibe `query` como argumento que se pasa como `...spread` a `.where()`. Para `find(null, options)` salta el `.where()` — compatible con el uso actual en `GET /users`.

### Patrones relevantes confirmados

- `authenticate` se aplica a nivel de router (`.use(authenticate)`), no por ruta. El Bloque 2 ya lo hizo así en `redirectRouterApi`. El Bloque 4 sigue el mismo patrón.
- `authorize('admin')` se aplica por ruta, no a nivel de router, porque la excepción es `GET /me` — accesible a cualquier usuario autenticado. El resto de rutas (`GET /`, `GET /:id`, `POST /`, `DELETE /:id`, `PATCH /:id`) requieren admin o tienen restricciones propias (PATCH: no-admin solo edita su propio perfil).
- La respuesta del `PATCH` devuelve `data.toPublic()` — igual que el handler existente, sin cambio.
- El spec (§4.3) indica que el `PATCH` handler valida inline con `schema.validate(req.body)` en lugar de usar `validatorHandler`. Esto es una desviación puntual justificada: el schema a usar depende de `req.user.role`, que solo está disponible en tiempo de ejecución, no en tiempo de configuración del router.

## Decisiones de diseño resueltas

### D-B4-1: `GET /api/v1/users` — acceso

**Decisión: `authorize('admin')` únicamente.**

Razón de seguridad: la lista expone emails, roles y membresía de grupos. No hay caso de uso legítimo para un usuario regular en v3. Si en versiones futuras se necesita un directorio, se creará un endpoint dedicado con proyección reducida (sin `role` ni `groups`).

### D-B4-2: `GET /api/v1/users/:id` — acceso a perfil ajeno

**Decisión: solo admins acceden a perfiles de otros usuarios.**

Los usuarios regulares usan `GET /me` para su propio perfil. No se implementa ownership check en `GET /:id` — la separación de responsabilidades es más limpia: `/:id` es admin-only, `/me` es para el usuario autenticado.

### D-B4-3: `PATCH /:id` — validación inline

**Decisión: validación inline del body con schema dinámico por rol.**

Mejora de mantenibilidad: extraer la selección del schema a `selectUpdateSchema(role)` en `user.schema.js` para que la lógica viva junto a los schemas. La validación en sí permanece en el handler.

### D-B4-4: `GET /me` — montaje antes de `GET /:id`

**Decisión: seguir el spec.** El orden de declaración en el router es:
1. `GET /me`
2. `GET /:id`

## Riesgos identificados

**R1 — Tests existentes de `user.route.api.test.js`**  
Los tests existentes en `src/api/users/routes/__test__/user.route.api.test.js` no incluyen JWT. Al añadir `authenticate` a nivel de router, todos los tests que no envíen `Authorization: Bearer <token>` recibirán 401 en lugar del status esperado. Los tests regresarán. Resolución: trabajo del test-engineer; usar el mock de `authenticate` con `x-test-user` header, igual que en `redirect.route.api.test.js`.

**R2 — `GET /me` lookup**  
Decisión: `req.user.userId` → `userService.findOne(userId)`. Lookup por clave primaria (O(1)), más eficiente que `getByEmail`. Robusto ante futuros cambios de email. `UserServices` hereda `findOne(id)` de `CrudService` — no requiere cambios en el servicio.

**R3 — Bug en constructor de `User` — `role` defaultea a `'user'`**  
Bug: `this.role = role || 'user'` en el constructor convierte `undefined` en `'user'`. En un PATCH sin `role` en el body, el handler construye `new User({ id, ...value })` donde `role` es `undefined`, y el constructor lo rellena con `'user'`. Firestore escribe `role: 'user'` sobre el perfil, incluyendo sobre admins.

Fix confirmado: mover el default de `role` al `createUserParser`. El constructor queda `this.role = role` (igual que `Group`). `cleanDocObject` en `updateUserParser` eliminará `role: undefined` antes de escribir en Firestore. Este fix se realiza en la Etapa 0, antes de la implementación del Bloque 4.

**Fix de congruencia de modelos:** Los constructores de `*.model.api.js` no deben aplicar defaults (`|| value`) a campos opcionales. Los defaults pertenecen a los parsers de creación. Este principio ya se aplicó a `Group` (R6 de Bloque 3). Se aplica igualmente a `User` como parte de la misma corrección (R3 de Bloque 4).

**R4 — `updateUserSchema` actual expone `auth`**  
El `updateUserSchema` actual permite actualizar el campo `auth` (tokens). El spec §4.3 no incluye `auth` en ninguno de los dos schemas nuevos. El split elimina esa posibilidad — potencialmente breaking change si algún cliente lo usaba. Sin embargo, dado que las rutas de usuarios eran públicas (sin autenticación), no hay clientes legítimos que dependan de este comportamiento. Riesgo bajo.

**R5 — `firstName`/`lastName` son `name` en el schema, no `firstName`/`lastName`**  
En `user.schema.js` los primitivos son `name = Joi.string().max(70).min(1)` y se reusan para ambos campos. El split mantiene este patrón — no hay cambio en la validación de esos campos.
