## Pasos

### §0.1 — Límite de 10 grupos en user.schema.js

El campo `groups` en `user.schema.js` se define en la línea 6 como `Joi.array().items(Joi.string())`. Esta definición se reutiliza en `createUserSchema` (línea 26) y `updateUserByAdminSchema` (línea 39). Sin un límite superior, un admin podría crear o actualizar un usuario con más de 10 grupos, lo que en v4 haría que la query `array-contains-any` de `GET /api/v1/redirects` supere el máximo de 10 valores que acepta Firestore (ver §1.4 del spec).

- [x] **Paso 1 — Añadir `.max(10)` a la definición base del campo `groups` en `src/api/users/schemas/user.schema.js`.**

  La variable `groups` en la línea 6 es la única definición del campo que se reutiliza en ambos schemas. Modificar esta línea aplica el límite simultáneamente en `createUserSchema` y `updateUserByAdminSchema`. No es necesario tocar ninguna otra línea del archivo. El mensaje de validación que Joi retornará automáticamente es `"groups" must contain less than or equal to 10 items`, que `validatorHandler` convierte en un `400 Bad Request`.

- [x] **Paso 2 — Añadir tests para el límite en `src/api/users/routes/__test__/user.route.api.test.js`.**

  Los tests deben cubrir:
  - `POST /api/v1/users` con un array `groups` de 11 elementos rechaza con 400.
  - `POST /api/v1/users` con un array `groups` de exactamente 10 elementos es aceptado (valor en el límite, no violación).
  - `PATCH /api/v1/users/:id` con un array `groups` de 11 elementos rechaza con 400.
  - `PATCH /api/v1/users/:id` con un array `groups` de exactamente 10 elementos es aceptado.

  El test de `PATCH` debe usar la ruta de admin (`req.user.role === 'admin'`) porque `updateUserSelfSchema` no incluye el campo `groups`.

- [x] **Paso 3 — Verificar que no se requieren cambios en `GroupService.update()`.**

  `GroupService.update()` sincroniza `User.groups` via `userService.update()` en un loop — este camino no pasa por el schema Joi de usuario (pasa por el service directamente). El spec aclara que la guardia de schema es para el input del cliente, no para la sincronización interna. El backend-engineer debe confirmar que `GroupService.update()` no puede producir arrays de más de 10 grupos como efecto colateral de su sync; si puede, documentarlo como riesgo en el commit pero no bloquearlo — es una decisión de diseño de Sprint 2 (§1.3).

### BUG-A — error handler responde HTTP 200 al servir páginas HTML

El `errorHandler` en `src/middleware/error.handler.js` llama a `res.sendFile()` directamente en las ramas 404 (línea 47) y 500/prod (línea 49) sin haber llamado a `res.status(statusCode)` antes. Express envía HTTP 200 por defecto cuando no se llama a `res.status()`. Un cliente HTTP que recibe una respuesta 200 no sabe que se produjo un error.

Los tests en `src/middleware/__test__/error.handler.test.js` actualmente verifican el comportamiento incorrecto: la línea 61 aserta `expect(mockRes.status).not.toHaveBeenCalled()` para el caso 500/prod, y la línea 94 aserta solo que `sendFile` fue llamado para 404, sin verificar que `status(404)` también fue llamado. Ambos tests deben actualizarse en el mismo commit del fix, no después: un test que verifica el comportamiento incorrecto como correcto es un riesgo mayor que no tener test.

- [ ] **Paso 4 — Corregir las ramas 404 y 500/prod en `src/middleware/error.handler.js`.**

  En la rama 404 (línea 47): encadenar `res.status(statusCode).sendFile(...)` en lugar de `res.sendFile(...)`.
  En la rama 500/prod (línea 49): ídem.
  La rama `else` (línea 51) ya usa `res.status(statusCode).json(...)` correctamente — no tocarla.

- [ ] **Paso 5 — Actualizar los tests en `src/middleware/__test__/error.handler.test.js`.**

  Los dos tests afectados son:
  - `'should serve the HTML error page for 500 errors in production'` (línea 55): cambiar `expect(mockRes.status).not.toHaveBeenCalled()` por `expect(mockRes.status).toHaveBeenCalledWith(500)`. Añadir `expect(mockRes.sendFile).toHaveBeenCalled()` si no está ya presente.
  - `'should respond with not found page for 404 status code'` (línea 91): añadir `expect(mockRes.status).toHaveBeenCalledWith(404)` para verificar que el status correcto fue enviado antes del HTML.

  Estos dos pasos (Paso 4 y Paso 5) deben ir en el mismo commit `[fix]`. No es aceptable un commit que corrija el código pero deje los tests asertar el comportamiento viejo.

### BUG-D — CrudService.find() no aplica orderBy cuando hay query

En `src/utils/crud.service.js`, el método `find()` aplica `orderBy('updated', 'desc')` por defecto solo cuando no hay query (líneas 74-75: `else if (!orderBy)`). Cuando se pasa una query, el bloque `else if` no se ejecuta y no hay ningún orden garantizado. La paginación con `offset` sobre una colección sin orden estable es no-determinista: el mismo offset puede retornar distintos documentos entre requests.

- [x] **Paso 6 — Corregir el bloque de ordenamiento en `CrudService.find()` en `src/utils/crud.service.js`.**

  La condición actual que aplica el orden por defecto es `else if (!orderBy)`, que excluye el caso `query && !orderBy`. El fix debe aplicar `orderBy('updated', 'desc')` siempre que `options.orderBy` no esté especificado, independientemente de si `query` tiene valor.

  La restricción de Firestore relevante: cuando se usa un filtro `where()`, Firestore requiere que el campo en `orderBy()` esté incluido en el índice compuesto. Los índices en `firestore.indexes.json` ya cubren `redirects/owner+updated` y `redirects/permission+updated`, que son las queries reales que `find()` recibe en producción. El backend-engineer debe verificar que el fix no introduce una query que no tenga índice para otras colecciones (users, groups) si `find()` se usa en ellas.

- [x] **Paso 7 — Añadir o actualizar tests en `src/utils/__test__/crud.service.test.js` (o donde vivan los tests de CrudService).**

  Los tests deben cubrir:
  - `find()` sin query aplica `orderBy('updated', 'desc')` — comportamiento ya existente, verificar que no regresiona.
  - `find()` con query y sin `options.orderBy` aplica `orderBy('updated', 'desc')` — este es el caso buggy que ahora debe pasar.
  - `find()` con query y con `options.orderBy` explícito usa el `orderBy` especificado, no el por defecto.

  Nota: el backend-engineer debe encontrar dónde viven actualmente los tests de `CrudService` (`grep -rn "CrudService\|crud.service" src/ --include="*.test.js"`) antes de escribir tests nuevos, para no crear un archivo duplicado.

### §1.1 — Singleton del cliente Firestore

`FireStoreAdapter` instancia `new Firestore.Firestore()` en su constructor (línea 9 de `src/lib/firestore.js`). Con tres colecciones activas (redirects, users, groups), hay tres instancias del SDK del cliente Firestore en memoria. Esto no produce un costo extra en Firestore (el SDK ya comparte el pool de conexiones internamente), pero bloquea la posibilidad de batch writes entre colecciones que se implementará en §1.3 de Sprint 2 — un `WriteBatch` debe crearse desde el mismo cliente que referencia los `docRef` incluidos en el batch.

El fix es extraer la instanciación del cliente a un módulo Node.js que exporta un singleton. Dado que `require()` en Node.js cachea el módulo en la primera carga, un archivo que solo hace `new Firestore.Firestore()` y lo exporta garantiza que todos los consumidores comparten la misma instancia.

- [ ] **Paso 8 — Crear `src/lib/firestore-client.js` como módulo singleton.**

  El archivo debe cumplir exactamente el contrato especificado en §1.1 del spec: importar `Firestore` del paquete `@google-cloud/firestore`, crear una instancia, y exportarla. El constructor no recibe argumentos — el SDK lee la configuración de las credenciales del entorno (`GOOGLE_APPLICATION_CREDENTIALS` o Application Default Credentials en GAE).

  El módulo no debe tener lógica adicional: sin logging, sin lazy initialization, sin parámetros de configuración. La simplicidad es un requisito — este módulo es infraestructura compartida.

- [ ] **Paso 9 — Actualizar `FireStoreAdapter` para importar el singleton en lugar de instanciar uno propio.**

  En `src/lib/firestore.js`, el constructor actualmente hace `this.db = new Firestore.Firestore()`. Debe cambiarse para importar `firestoreClient` desde `./firestore-client` y asignarlo a `this.db`. La API pública de `FireStoreAdapter` no cambia — `this.db`, `this.collection`, y todos los métodos siguen siendo los mismos. El cambio es interno al constructor.

  El import de `@google-cloud/firestore` en `firestore.js` sigue siendo necesario por el uso de `Firestore.Timestamp` en los métodos `create()` y `update()`. No eliminarlo.

- [ ] **Paso 10 — Actualizar los tests de `FireStoreAdapter` en `src/lib/__test__/firestore.test.js`.**

  Los tests actuales mockean `@google-cloud/firestore` para evitar llamadas reales a Firestore. Con el cambio, `FireStoreAdapter` ya no instancia el cliente directamente — lo importa desde `firestore-client.js`. El mock debe ajustarse para que `jest.mock('./firestore-client')` (o la ruta correspondiente desde el test) retorne un objeto que simule el cliente Firestore en lugar de mockear el constructor de `Firestore.Firestore`.

  El backend-engineer debe revisar la estructura actual del mock en `firestore.test.js` antes de proceder — el ajuste exacto depende de cómo el test mockea el módulo actualmente. El criterio de aceptación es que todos los tests de `FireStoreAdapter` sigan pasando sin cambios de comportamiento.

## Orden de implementación recomendado

**Orden:** §0.1 → BUG-A → BUG-D → §1.1

**Razones:**

1. **§0.1 primero** porque es el cambio de menor superficie (una línea en un archivo, tests en un archivo de test ya existente) y cierra una dependencia de seguridad de Firestore documentada en el spec. Completarlo primero permite a los sprints subsiguientes asumir que el invariante de "máximo 10 grupos por usuario" está garantizado.

2. **BUG-A segundo** porque corrige un comportamiento observable externamente (respuesta HTTP 200 en lugar de 404/500) y sus tests son los únicos que actualmente asertan comportamiento incorrecto como correcto. Resolverlo antes de BUG-D evita tener dos tests "con fallo intencional" simultáneos que podrían confundir el estado del suite.

3. **BUG-D tercero** porque es independiente de los anteriores y su fix está contenido en un solo método de `CrudService`. El riesgo de que el fix introduzca queries sin índice en colecciones distintas a `redirects` debe resolverse antes de que §1.1 cambie la arquitectura del cliente Firestore, para evitar depurar dos cambios simultáneos.

4. **§1.1 último** en Sprint 1 porque, aunque es un prerequisito de §1.3 (Sprint 2), no desbloquea nada dentro de Sprint 1. Hacerlo último permite que los tres fixes anteriores tengan el suite en verde antes de introducir el cambio estructural del singleton. Los tests de `FireStoreAdapter` requieren el ajuste de mocks más cuidadoso de todas las unidades de Sprint 1 — conviene tener el resto del suite estable antes de abordarlos.

**Dependencia entre §1.1 y Sprint 2:** el batch write de §1.3 requiere acceso al cliente Firestore compartido. El módulo `src/lib/firestore-client.js` creado en §1.1 es exactamente lo que `GroupService` importará en §1.3 para construir el `WriteBatch`. Sin §1.1 completado, §1.3 no puede implementarse. No iniciar Sprint 2 sin §1.1 en verde.

## Notas para el backend-engineer

**BUG-A — Commits de fix y test en secuencia, no en paralelo:**

El Paso 4 (fix) y el Paso 5 (actualización de tests) van en el mismo commit `[fix]`. El motivo: los tests en `error.handler.test.js` actualmente asertan el comportamiento incorrecto. Si el Paso 4 va en un commit sin el Paso 5, el suite quedará rojo (los tests "correctos" fallan porque el código ya no se comporta como ellos esperan). Un commit que rompe el suite no es aceptable — la rama `dev` debe pasar siempre.

**BUG-D — Verificar impacto en otras colecciones antes de hacer el fix:**

`find()` se usa con queries en `RedirectServiceApi.getByPath()` y en los handlers de GET. Antes de modificar el `else if`, el backend-engineer debe confirmar qué colecciones llaman a `find()` con query y verificar que los índices compuestos necesarios existen en `firestore.indexes.json`. La solución más segura es añadir el `orderBy` por defecto solo cuando no hay `orderBy` explícito — la condición cambia de `else if (!orderBy)` a una guardia que aplique el default independientemente de si hay query.

**§1.1 — El Timestamp sigue viniendo de `@google-cloud/firestore`:**

El import `const Firestore = require('@google-cloud/firestore')` en `firestore.js` no se elimina porque `Firestore.Timestamp` se usa en `create()` y `update()`. La confusión posible es que se elimine el import pensando que el singleton ya lo trae. No — el singleton exporta una instancia del cliente, no el módulo completo.
