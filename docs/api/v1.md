## REST API v1 — 1kg.me

Base URL: `https://1kg.me/api/v1`

All endpoints return JSON. Errors follow the [Boom format](#errors).

---

## Authentication

### OAuth2 + JWT flow

1. Send the user to `GET /api/v1/auth/google`. The browser is redirected to Google's consent screen.
2. After the user grants access, Google redirects to the callback URL. The server looks up the Google account email in Firestore.
   - If the email is not registered, the server returns `401`. Users are created exclusively by admins — there is no self-registration.
3. On success, the callback returns a JWT the client must store and send in every subsequent request.

### `GET /api/v1/auth/google`

Initiates the Google OAuth2 flow. Redirects the browser to Google's authorization page.

No request body. No JSON response — this endpoint triggers a browser redirect.

### `GET /api/v1/auth/google/callback`

OAuth2 callback. The server exchanges the authorization code for a Google token, verifies the email is registered, and returns a signed JWT.

**Success response — 200**

```json
{
  "message": "login successful",
  "data": {
    "token": "<jwt>",
    "user": {
      "id": "abc123",
      "email": "user@example.com",
      "firstName": "Jane",
      "lastName": "Doe",
      "groups": ["fc", "cs"],
      "role": "user",
      "created": "2026-01-15T10:00:00.000Z",
      "updated": "2026-06-01T08:30:00.000Z"
    }
  }
}
```

**Error responses**

| Code | Condition |
|------|-----------|
| 401  | Email not registered in Firestore |

### JWT

| Property | Value |
|----------|-------|
| Algorithm | HS256 |
| Default TTL | 2 hours (configurable via `JWT_TTL`) |
| Payload fields | `userId`, `email`, `role`, `groups` |

The token payload reflects the user's state **at login time**. Group membership changes take effect at the next login.

### Sending the token

Include the token in every protected request:

```
Authorization: Bearer <token>
```

Missing or invalid tokens return `401`.

### API Key authentication

API Keys are an alternative bearer credential. A token with the prefix `sk_1kg_` is treated as an API Key — the server hashes it with SHA-256 and looks it up via `ApiKeyService.findByHash`.

```
Authorization: Bearer sk_1kg_<32-character base62 token>
```

**Restrictions:**

- API Keys are accepted **only on `/api/v1/redirects`** endpoints. Every endpoint on `/api/v1/users` and `/api/v1/groups` (including `/api/v1/users/me/api-keys`) rejects API Key bearer tokens with `403`.
- Each API Key carries a list of **scopes**. The required scope is enforced per-route — see [Redirects](#redirects) for which scope applies to each endpoint.
- A revoked or expired key returns `401`. After revocation, the key may remain valid for up to 30 seconds due to server-side caching; same-instance invalidation is best-effort.

## Redirects

Base path: `/api/v1/redirects`

All endpoints require authentication — either a valid JWT or an API Key with the appropriate scope. The `owner` field is always derived from the authenticated user — it cannot be set or overridden in the request body.

### `GET /api/v1/redirects`

Required scope (API Key): `read:redirects`

Returns redirects visible to the authenticated user.

- Admins receive all redirects in the system.
- Regular users receive only redirects they own or that have a `read:{slug}` permission entry matching one of their groups.

**Query parameters**

| Parameter | Type | Description |
|-----------|------|-------------|
| `orderBy` | string | Field name to sort by. Prefix with `-` for descending (e.g. `-created`). Optional. |
| `offset` | integer ≥ 1 | Number of records to skip. Optional. |
| `limit` | integer ≥ 1 | Maximum records to return. Optional. |

**Success response — 200**

```json
{
  "message": "redirects retrieved",
  "data": [
    {
      "id": "xyz789",
      "path": "/fc/seminar",
      "url": "https://example.com/seminar-2026",
      "owner": "user@example.com",
      "permission": ["read:fc"],
      "categories": ["events"],
      "created": "2026-03-10T12:00:00.000Z",
      "updated": "2026-03-10T12:00:00.000Z"
    }
  ]
}
```

---

### `GET /api/v1/redirects/:id`

Required scope (API Key): `read:redirects`

Returns a single redirect by Firestore document ID.

Access is granted if the requester is an admin, the owner of the redirect, or belongs to a group listed in `redirect.permission` with a `read` scope.

**Success response — 200**

```json
{
  "message": "redirect retrieved",
  "data": {
    "id": "xyz789",
    "path": "/fc/seminar",
    "url": "https://example.com/seminar-2026",
    "owner": "user@example.com",
    "permission": ["read:fc"],
    "categories": ["events"],
    "created": "2026-03-10T12:00:00.000Z",
    "updated": "2026-03-10T12:00:00.000Z"
  }
}
```

**Error responses**

| Code | Condition |
|------|-----------|
| 403  | Authenticated user has no read access to this redirect |
| 404  | Redirect not found |

---

### `POST /api/v1/redirects`

Required scope (API Key): `write:redirects`

Creates a new redirect.

**Namespace rules**

- Regular users must supply `group`. The `group` slug must match one of the slugs in the user's JWT `groups` claim. The server builds the full path as `/{group}/{path}`.
- Admins may omit `group`. The full path will be `/{path}`. Admins can also supply `group` to create under a group namespace.

**Request body**

| Field | Type | Required | Validation |
|-------|------|----------|------------|
| `path` | string | Yes | Lowercase alphanumeric and hyphens, segments separated by `/`. No leading slash. Pattern: `^[a-z0-9][a-z0-9-]*(\/[a-z0-9][a-z0-9-]*)*$` |
| `url` | string | Yes | Valid URI |
| `group` | string | Required for non-admins | Lowercase alphanumeric and hyphens. Pattern: `^[a-z0-9-]+$` |
| `permission` | string[] | No | Each entry must match `^(read\|edit\|delete):[a-z0-9-]+$` |
| `categories` | string[] | No | Array of arbitrary strings |

The `owner` field is set from the authenticated user's email — it is never accepted in the body.

**Example request**

```json
{
  "group": "fc",
  "path": "seminar",
  "url": "https://example.com/seminar-2026",
  "permission": ["read:fc"],
  "categories": ["events"]
}
```

This produces the stored path `/fc/seminar`, accessible at `1kg.me/fc/seminar`.

**Success response — 201**

```json
{
  "message": "redirect created",
  "data": {
    "id": "xyz789",
    "path": "/fc/seminar",
    "url": "https://example.com/seminar-2026",
    "owner": "user@example.com",
    "permission": ["read:fc"],
    "categories": ["events"],
    "created": "2026-03-10T12:00:00.000Z",
    "updated": "2026-03-10T12:00:00.000Z"
  }
}
```

**Error responses**

| Code | Condition |
|------|-----------|
| 400  | Validation error, or path already taken |
| 403  | Non-admin omitted `group`, or `group` does not match the user's group membership |

---

### `PATCH /api/v1/redirects/:id`

Required scope (API Key): `write:redirects`

Updates a redirect. Only the owner or an admin can modify a redirect.

**Request body** — all fields optional

| Field | Type | Validation |
|-------|------|------------|
| `path` | string | Same pattern as `POST` — no leading slash |
| `url` | string | Valid URI |
| `permission` | string[] | Same format as `POST` |
| `categories` | string[] | Array of strings |

Fields absent from the body are left unchanged. `owner` cannot be changed.

**Success response — 200**

```json
{
  "message": "redirect updated",
  "data": {
    "id": "xyz789",
    "path": "/fc/seminar",
    "url": "https://example.com/seminar-new",
    "owner": "user@example.com",
    "permission": ["read:fc", "read:cs"],
    "categories": ["events"],
    "created": "2026-03-10T12:00:00.000Z",
    "updated": "2026-06-09T09:00:00.000Z"
  }
}
```

**Error responses**

| Code | Condition |
|------|-----------|
| 400  | Validation error |
| 403  | Requester is not the owner and not an admin |
| 404  | Redirect not found |

---

### `DELETE /api/v1/redirects/:id`

Required scope (API Key): `write:redirects`

Deletes a redirect. Only the owner or an admin can delete a redirect.

**Success response — 200**

```json
{
  "message": "redirect deleted",
  "data": "xyz789"
}
```

The `data` field contains the deleted document ID.

**Error responses**

| Code | Condition |
|------|-----------|
| 403  | Requester is not the owner and not an admin |
| 404  | Redirect not found |

### `PATCH /api/v1/users/:id`

Updates a user. Admins can edit any user. Regular users can only edit their own profile.

The set of editable fields differs by role:

| Field | Admin | Regular user |
|-------|-------|--------------|
| `firstName` | Yes | Yes |
| `lastName` | Yes | Yes |
| `role` | Yes | No |
| `groups` | Yes | No |

Updating `groups` on a user does not affect `Group.users` — group membership is the authoritative source. Use `PATCH /api/v1/groups/:id` to manage group membership, which keeps both sides in sync.

**Example request (admin)**

```json
{
  "role": "admin",
  "groups": ["fc", "cs"]
}
```

**Example request (regular user)**

```json
{
  "firstName": "Jane",
  "lastName": "Roe"
}
```

**Success response — 200**

```json
{
  "message": "user updated",
  "data": {
    "id": "abc123",
    "email": "user@example.com",
    "firstName": "Jane",
    "lastName": "Roe",
    "groups": ["fc"],
    "role": "user",
    "created": "2026-01-15T10:00:00.000Z",
    "updated": "2026-06-09T11:00:00.000Z"
  }
}
```

**Error responses**

| Code | Condition |
|------|-----------|
| 400  | Validation error, or attempt to set a field not allowed for the requester's role |
| 403  | Regular user attempting to edit another user's profile |
| 404  | User not found |

## API Keys

Base path: `/api/v1/users/me/api-keys`

All endpoints require a valid JWT. API Key bearer tokens are rejected with `403` — managing API keys requires a full session. Creating or revoking keys cannot be done using an API Key credential.

API Keys are scoped credentials that allow machine clients to authenticate against `/api/v1/redirects` without a JWT. The plaintext token is returned **only once** at creation time and is not stored — only its SHA-256 hash is kept in Firestore.

### `GET /api/v1/users/me/api-keys`

Returns all API keys for the authenticated user. The response never includes the plaintext token or its hash.

**Success response — 200**

```json
{
  "message": "api keys retrieved",
  "data": [
    {
      "id": "key001",
      "name": "CI pipeline",
      "prefix": "sk_1kg_",
      "scopes": ["read:redirects", "write:redirects"],
      "expiresAt": null,
      "createdAt": "2026-06-10T09:00:00.000Z",
      "lastUsedAt": "2026-06-20T14:30:00.000Z",
      "active": true
    }
  ]
}
```

---

### `POST /api/v1/users/me/api-keys`

Creates a new API key for the authenticated user.

**Limits:**
- A user may have at most 10 active keys. Creating an 11th returns `400`.

**Request body**

| Field | Type | Required | Validation |
|-------|------|----------|------------|
| `name` | string | Yes | Maximum 100 characters |
| `scopes` | string[] | Yes | At least one scope. Valid values: `read:redirects`, `write:redirects`, `read:users`, `write:users`, `read:groups`, `write:groups` |
| `expiresAt` | string (ISO 8601) | No | Future date. `null` means no expiry. |

Admin-only scopes (`read:users`, `write:users`, `read:groups`, `write:groups`) can only be requested by users with `role: "admin"`. Non-admin users requesting these scopes receive `403`.

**Example request**

```json
{
  "name": "CI pipeline",
  "scopes": ["read:redirects", "write:redirects"],
  "expiresAt": null
}
```

**Success response — 201**

The `token` field is returned **only in this response**. Store it immediately — it cannot be retrieved again.

```json
{
  "message": "api key created",
  "data": {
    "id": "key001",
    "name": "CI pipeline",
    "prefix": "sk_1kg_",
    "scopes": ["read:redirects", "write:redirects"],
    "expiresAt": null,
    "createdAt": "2026-06-10T09:00:00.000Z",
    "lastUsedAt": null,
    "active": true,
    "token": "sk_1kg_A7bC3dE9fG2hI5jK8lM1nO4pQ6rS0tU"
  }
}
```

**Error responses**

| Code | Condition |
|------|-----------|
| 400  | Validation error, or 10-active-key limit reached |
| 403  | Non-admin user requested an admin-only scope |

---

### `DELETE /api/v1/users/me/api-keys/:keyId`

Revokes an API key. The key is marked inactive in Firestore. After revocation, the key may remain valid for up to 30 seconds due to server-side caching.

**Success response — 200**

```json
{
  "message": "api key revoked",
  "data": { "id": "key001" }
}
```

**Error responses**

| Code | Condition |
|------|-----------|
| 404  | Key not found or does not belong to the authenticated user |

## Groups

Base path: `/api/v1/groups`

All endpoints require a valid JWT. API Key bearer tokens are rejected with `403`. The `slug` field is immutable after a group is created.

When a group's `users` array is changed via `PATCH`, the server automatically syncs `User.groups` for every added or removed member. The two collections are kept consistent by the service layer — callers do not need to update users separately.

### `GET /api/v1/groups`

Returns groups visible to the authenticated user.

- Admins receive all groups.
- Regular users receive only groups whose `slug` appears in their JWT `groups` claim.

**Query parameters**

| Parameter | Type | Description |
|-----------|------|-------------|
| `orderBy` | string | Field name to sort by. Prefix with `-` for descending. Optional. |
| `offset` | integer ≥ 0 | Number of records to skip. Optional. |
| `limit` | integer ≥ 1 | Maximum records to return. Optional. |

**Success response — 200**

```json
{
  "message": "groups retrieved",
  "data": [
    {
      "id": "grp001",
      "name": "Facultad de Ciencias",
      "slug": "fc",
      "users": ["abc123", "def456"],
      "created": "2026-01-01T00:00:00.000Z",
      "updated": "2026-01-01T00:00:00.000Z"
    }
  ]
}
```

Regular users with no groups receive an empty array.

---

## Errors

All errors use the Boom format:

```json
{
  "statusCode": 400,
  "error": "Bad Request",
  "message": "\"path\" is required"
}
```

In development (`NODE_ENV=development`) the response also includes a `stack` field.

**Common status codes**

| Code | Meaning |
|------|---------|
| 400 | Validation failed, duplicate key, or invalid input |
| 401 | Missing token, invalid token, email not registered, or API key revoked/expired |
| 403 | Authenticated but insufficient permissions |
| 404 | Resource not found |
| 500 | Internal server error |
